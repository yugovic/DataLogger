# Deep Insight: 車両スペックカード（トレカ型）のユーザ目線での課題と改善

- 日付: 2026-07-03
- 元の要望: "このコンテンツにおける車両カードについて、ユーザ目線で課題や改善点、もっとこうすれば良いということについて考えてみてください"
- 対象実装: `src/components/vehicle/SpecCard.tsx`（63:88 トレカ・表裏フリップ）＋利用箇所5つ
  （車両一覧 / セットアップ比較 / 共有カード・詳細 / 公開ランディング / 共有画像パイプライン）

## 0. 深掘り（Deepening）

### 反復プロセス
- Step 1（名詞→動詞）: 「車両カードが欲しい/良くしたい」→「**愛車の仕様と取り組みを、一枚で伝えたい**」
- Step 2: なぜ一枚で伝えたいのか？→「走行会やSNSで車の話をするたびに、仕様を口頭で説明し直すのは面倒で、伝わりきらないから」
  （根拠: 公開LPの訴求文言「スペックカードで愛車の仕様を共有」、共有画像パイプライン=SNS前提の設計）
- Step 3: なぜ伝えたいのか？→「車にかけた手間とお金、走りへの取り組みを**仲間に認めてもらいたい**から。カードはその『名刺』であり『戦績表』」
  （根拠: context.md P3ビギナーの承認欲求、P4売り手の「実績の可視化=ブランディング」動機）
- 到達した真因（仮説・否定形）:
  **「愛車と自分の取り組みが、"ただのデータ行"として扱われたくない。毎回説明し直す手間から解放されたい」**

### ユーザーへの確認
- 「カードの本当の役割は『仕様の表示』ではなく、**ドライバーの誇りを一枚に封じ込めた"名刺 兼 戦績カード"**ではありませんか？
  そうであれば、現状のカードには『戦績（ラップ）』と『成長（変化）』が欠けています。」

## 1. 動詞化（Need Statement）

「仕様を表示する」ではなく:
- **誇示する**: 愛車の仕様・取り組みを、見栄えのする一枚で見せびらかしたい
- **証明する**: 申告値だけでなく、走った証拠（ラップ・証憑）で裏付けたい
- **育てる**: 記録を重ねるほどカードが充実していく実感を得たい

## 3. ペルソナ適合

- **P3 ビギナー（最適合）**: カードは「走りっぱなしで何も残らない」への回答。共有画像→SNS→流入の獲得チャネルでもある
- **P4 売り手**: カード=ポートフォリオの最小単位。「申告値」のままでは実績証明にならない点が未充足
- **P1 中級者**: 比較ビュー・共有データ閲覧時に「前提条件の読み取り装置」として使う。compact変種の情報量が鍵
- 妥当性: 高。カードは記録動機（NSM）と獲得（公開LP）の両方に効く数少ない資産

## 4. 現状の課題棚卸し（ユーザ目線）

### A. 気づけない・分からない（発見性）
1. **フリップが発見されない**: 裏面（改造明細）への唯一の導線はフッターの小さな「⇄」。初見の公開LP訪問者はまずカードが裏返ることに気づかない
2. **改造度カラーの意味が伝わらない**: 「色=情報」（グレー/スカイ/バイオレット/ゴールド）が設計意図だが、凡例・ツールチップがどこにもない。初見者にはただの配色
3. **compact変種の「3カテゴリ改造」が抽象的**: 何を弄っているのか一切分からず、クリックへの誘引にもなっていない

### B. 寂しい・もったいない（空状態と魅力）
4. **データが薄い車両のカードが寂しい**: タイヤ区分・出力が未入力だとピルが1つ、写真なしのNORMAL車はグレー一色。null保存方針は正しいが、「入力すればカードが充実する」への誘導が無い（所有者ビュー限定のゴーストピル等の余地）
5. **公開ランディングのカードは常に写真なし**: スナップショットに写真を含めない方針のため、獲得チャネルの顔となるカードが常にゴーストタイポ版。opt-in での写真公開は検討余地あり
6. **「戦績」が無い**: ベストラップ・走行回数・ホームサーキットなど「走った証」がカードに一切ない。トレカなのに攻撃力欄が空。アプリ内には bestLap・ロガー証憑データが既に存在する

### C. 信頼・整合性
7. **「申告値」止まり**: 事業の堀は「証憑付きデータ」なのに、カードは自己申告のみ。ロガー証憑がある車両にはバッジで差を付けられる（売り手P4の実績証明にも直結）
8. **保存画像と画面の不一致リスク**: 画面のカードは表裏があるが、保存/共有される画像に裏面（改造明細）が含まれるかはユーザーから見えない。「見えているものが保存される」期待とのズレ

### D. 品質の細部
9. **アクセシビリティ**: 非表示面が `aria-hidden` されておらず、スクリーンリーダーは表裏両方を読み上げる。`prefers-reduced-motion` 未対応（回転アニメ固定）
10. **長い車名・ゴーストタイポの破綻**: `text-4xl` 固定のため長いモデル名で窮屈。line-clamp はあるが視覚的に美しくないケースが残る
11. **ダークモードとの馴染み**: 表面はライト固定（トレカとして意図的）だが、ダークUIの中で浮く場面がある。カード外周の処理でなじませ余地

## 5. 品質分類（Kano参照・推定）

| 分類 | 項目 |
|---|---|
| Must-be | 欠損時に破綻しない表示 / 保存画像と画面表示の一致 / モバイルでの視認性 / a11y基本対応（9） |
| Performance | 情報の充実度（戦績・写真）/ フリップ・色の分かりやすさ（1,2,3）/ 空状態からの入力誘導（4） |
| Delighter | ロガー証憑バッジ（7）/ ベストラップ刻印（6）/ カードが記録とともに「育つ」演出 |

## 6. 魅力品質の仮説（要検証）

- **仮説A: ベストラップ刻印＋ロガー証憑バッジ**
  カード表面に「BEST LAP（サーキット名つき）」を刻印し、ロガー計測由来なら証憑バッジを付ける。
  根拠: 深掘りの真因「名刺兼・戦績カード」。P4の実績証明ジョブ、P3の承認ジョブに直撃。
  NSM寄与: 「走って記録するとカードが強くなる」= 記録の内発的動機。公開LPのカードにも戦績が載り獲得力が上がる。
  既存資産: setups の bestLap / 共有パイプラインの hasLoggerEvidence で実現可能。

- **仮説B: カードが「育つ」**
  改造申告の追加・ベスト更新でカードに Ver. 表記（例: Ver.3 / 2026.07）を刻み、ビルドジャーナルと連動。
  更新時に軽いフリップ演出。根拠: テーマ「成長していく」・P3の成長実感ジョブ。
  NSM寄与: 更新のたびに「カードを見たくなる→共有したくなる」ループ。

- **仮説C: フリップ発見性のマイクロインタラクション**
  初回表示時に一度だけ小さく「めくれかけ」のヒントアニメ（reduced-motion 尊重）＋ホバー時の角めくれ。
  根拠: 課題1。裏面（改造明細）こそカードの本体情報なのに到達率が低い懸念。
  NSM寄与: 間接（共有体験の質）。実装コスト極小。

## 7. 意思決定と実装骨子

### 推奨案
**仮説A（ベストラップ刻印＋証憑バッジ）を本命**とし、同時に低コストの品質改善（課題1,2,9=フリップ発見性・色凡例ツールチップ・a11y）を抱き合わせる。
理由: カードの真因（戦績カード）に最短で応え、既存データのみで実現でき、公開LP=獲得チャネルの訴求力を直接高めるため。仮説Bは A の土台の上で次段。

### 実装範囲
- 最小: 車両に紐づく自己ベスト（サーキット名＋タイム）を表面フッター上に1行刻印。ロガー証憑があれば小バッジ。未走行なら非表示（偽データ禁止の原則どおり null は出さない）
- 拡張: サーキット別ベストを裏面に一覧 / 仮説B（Ver.表記）/ 写真の opt-in 公開
- 同梱する品質修正: 非表示面の aria-hidden 切替、prefers-reduced-motion、フリップヒント、改造度カラーのツールチップ凡例

### 技術概略
- `buildSpecCardView` に `bestRecord: { circuit, lapTime, hasLoggerEvidence } | null` を追加
- 所有者ビュー: setups から車両IDで自己ベストを導出（既存 bestLap フィールド）。共有スナップショット: 生成時に焼き込み（スナップショット方針と整合）
- shareImage.ts の描画にも同項目を追加（画面と画像の一致）

### UX 5段階モデルでの位置づけ
- 仮説A: 要件（Scope）〜骨格（Skeleton）の拡張
- 品質修正群: 表層（Surface）＋骨格（Skeleton）

### 受け入れ基準（テスト種）
- [ ] ベスト記録がある車両のカード表面にサーキット名＋タイムが表示される（unit: specCardView）
- [ ] ベスト記録が無い場合、該当行が一切表示されない（null を 0 やダミーで埋めない）（unit）
- [ ] ロガー証憑がある場合のみ証憑バッジが表示される（unit）
- [ ] 共有画像（PNG）にも同じベスト記録が描画される（integration: shareImage）
- [ ] 公開スナップショットに焼き込まれたベスト記録が公開LPで表示される（integration）
- [ ] 非表示面が aria-hidden され、スクリーンリーダーで二重読み上げされない（component test）
- [ ] prefers-reduced-motion 環境でフリップが即時切替になる（component test）
- [ ] 改造度カラーの意味がツールチップ等で確認できる（component test）

## ユーザーへの確認事項

1. 深掘り仮説の妥当性: カードの役割は「名刺 兼 戦績カード」という理解で合っているか
2. 仮説A のベストラップは「自己申告のタイムも載せる」か「ロガー証憑があるものだけ載せる」か（データ品質方針との兼ね合い）
3. 写真の opt-in 公開（課題5）はプライバシー方針的に許容するか
4. 着手順: 推奨どおり「仮説A＋品質修正群」で良いか、先に低コストの品質修正のみ行うか
