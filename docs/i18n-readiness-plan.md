# VELOCITY LOGGER 多言語対応準備計画

作成日: 2026-07-19  
対象リポジトリ: CarSetup6

## 1. 目的

VELOCITY LOGGER を、日本語を既定言語として維持しながら英語でも利用できる構造へ移行する。初期対応言語は `ja-JP` と `en` とし、追加言語を画面ごとの再設計なしで増やせることを目標にする。

この計画で扱う「多言語対応」は、画面文言の翻訳だけではない。次も対象に含める。

- 日付、時刻、数値、通貨、単位、複数形の表示
- Ant Design の標準コンポーネント文言
- バリデーション、トースト、空状態、確認ダイアログ
- CSV、共有カード、Canvas 生成画像、公開共有画面
- Firestore に保存される列挙値と、過去データの互換性
- HTML、PWA manifest、アクセシビリティ属性、OGP

単位系の切替（kPa/psi、km/mile、°C/°F）は翻訳とは別機能として扱う。初期リリースでは現在の保存単位を変えず、表示形式だけをロケール対応する。

### 保存データに関する重要な発見

1. `WeatherType` は現在 `'晴れ' | '曇り' | 'ウェット' | 'フルウェット'` であり、日本語がデータ値になっている。スキーマ、ハイライト判定、ダッシュボード、共有カードにも同じ値への依存がある。
2. `sessionType`、改造カテゴリ、タイヤ区分、調整グループ、調整位置などは英語の安定コードを持っているが、表示ラベルが型ファイルやドメインヘルパーに直書きされている。
3. `SetupAdjustmentDefinition.label`、`helpText`、`options`、`unit` と、保存時の `SetupAdjustmentValue.label` はユーザー作成コンテンツである。原則として自動翻訳せず、入力された言語のまま表示する。
4. `Track.id` は安定識別子だが、セットアップ本体の `circuit` は表示名・自由入力文字列として保存されている。既存値を一括で翻訳名に置換してはいけない。
5. `costJPY` と円表示は日本市場を前提にしている。翻訳時も保存通貨を JPY のまま扱い、別通貨への換算とは分離する。

## 3. 採用する基本設計

### 3.1 ロケール決定順

表示言語は次の優先順で決定する。

1. 公開共有 URL の明示指定（初期案: `?lang=en`）
2. ログインユーザーの `users/{uid}.locale`
3. 端末内に保存した直近設定 `velocity-logger.locale`
4. `navigator.languages`
5. 既定値 `ja-JP`

認証済みユーザーが言語を変えた場合は Firestore と端末の両方へ保存する。未ログイン時は端末だけに保存し、ログイン後にプロフィール設定を優先する。

### 3.2 ライブラリと配置

`i18next` と `react-i18next` を導入済み。現在の翻訳リソースは型検査とキー整合テストを行いやすい `src/i18n/resources.ts` に集約し、対象言語が増えて分割が必要になった段階で次の名前空間構造へ移す。

```text
src/i18n/
  index.ts
  locale.ts
  formatters.ts
  resources/
    ja-JP/
      common.json
      auth.json
      setup.json
      vehicle.json
      telemetry.json
      share.json
      errors.json
    en/
      common.json
      auth.json
      setup.json
      vehicle.json
      telemetry.json
      share.json
      errors.json
```

名前空間は画面ではなく業務領域で分ける。翻訳キーは原文をそのままキーにせず、`setup.weather.sunny` のような意味ベースの安定キーにする。

```tsx
t('setup:weather.sunny')
t('common:itemCount', { count })
t('setup:save.success', { vehicle: vehicleName })
```

文中の語を JSX や文字列結合で組み立てない。語順が変わる言語に備え、1 文を 1 翻訳単位にして補間する。複数形は i18next の `count` 規則を使う。

### 3.3 Provider と UI ライブラリ

アプリ最上位にロケール Provider を置き、同じロケールから次を更新する。

- i18next の現在言語
- Ant Design `ConfigProvider` の `ja_JP` / `en_US`
- `document.documentElement.lang`
- `document.title`
- 日時・数値フォーマッター

言語選択は既存の Header 設定モーダル内に置く。通常操作の主導線を増やさないため、ヘッダー上に常設の大きな切替ボタンは追加しない。未ログイン画面と公開共有画面では、必要時に開ける小さな言語メニューを用意する。

### 3.4 日付、数値、通貨、単位

ロケールを直書きした `toLocaleDateString('ja-JP')` や `Intl.DateTimeFormat('ja-JP')` は共通フォーマッターへ集約する。

必要な API は最低限、次のとおり。

- `formatDate(date, style)`
- `formatDateTime(date, style)`
- `formatNumber(value, options)`
- `formatInteger(value)`
- `formatCurrency(value, currency)`
- `formatUnit(value, unit)`
- `formatRelativeTime(value, unit)`

保存値は変更しない。たとえば空気圧は引き続き kPa、距離は km、温度は °C、費用は JPY とする。将来単位系を追加するときは、入力・表示時だけ変換し、保存時は既定単位へ正規化する。

## 4. データモデル移行

### 4.1 日本語列挙値を安定コードへ変更

天候は次のコードへ移行する。

| 旧保存値 | 新コード | 日本語表示 | 英語表示 |
| --- | --- | --- | --- |
| `晴れ` | `sunny` | 晴れ | Sunny |
| `曇り` | `cloudy` | 曇り | Cloudy |
| `ウェット` | `wet` | ウェット | Wet |
| `フルウェット` | `full_wet` | フルウェット | Full wet |

移行は破壊的な一括更新にせず、次の順で行う。

1. 読取時に旧値と新コードの両方を受け付け、新コードへ正規化する。
2. 新規保存と更新保存は新コードだけを書き込む。
3. 判定ロジック、グラフ集計、共有表示は正規化後のコードだけを見る。
4. テスト用フィクスチャに旧値ケースを残し、後方互換を検証する。
5. 十分な移行期間後に、必要なら管理スクリプトで既存データを更新する。

Firestore ルールやクエリが列挙値を参照している場合は、旧値と新値の併存期間を考慮する。

### 4.2 翻訳対象と非対象を区別する

| データ | 扱い |
| --- | --- |
| UI 固定文言、標準選択肢、検証メッセージ | 翻訳キーで管理 |
| `sessionType` などの安定コード | 保存値は不変、表示だけ翻訳 |
| サーキットの公式名称 | `Track.id` に対するロケール別表示名を用意 |
| 自由入力の `circuit` | 入力値をそのまま表示 |
| 車種、メーカー、パーツ名、タイヤ製品名 | ユーザーまたはカタログの原文を維持 |
| 動的調整項目のユーザー作成ラベル | 自動翻訳しない |
| 組み込み調整項目 | `labelKey` を持たせて翻訳可能にする |
| 過去記録のラベルスナップショット | 当時の原文を維持し、勝手に置換しない |
| ユーザーメモ、学び、所見 | 自動翻訳しない |

組み込み項目とユーザー作成項目を同じ型で扱う場合は、`label` を残したまま任意の `labelKey` と `source: 'builtin' | 'user'` を追加する。表示時は `labelKey` があり翻訳が存在すれば翻訳を使い、なければ `label` へフォールバックする。

### 4.3 エラーをコード化する

サービス層と Zod スキーマが日本語の完成文を返す構造をやめ、UI が翻訳できるエラーコードへ寄せる。

```ts
type AppErrorCode =
  | 'setup.vehicle_not_found'
  | 'setup.vehicle_forbidden'
  | 'validation.required'
  | 'validation.out_of_range';
```

エラーには `code`、`field`、補間用 `params`、ログ用の `cause` を持たせる。Firebase など外部由来の例外はユーザーへ直接表示せず、既知コードへマッピングする。ログは翻訳せず、安定コードで検索できるようにする。

## 5. 画面・出力別の要件

### 認証、オンボーディング、設定

- 初回表示前に言語を決定し、日本語が一瞬表示されてから英語へ変わる現象を防ぐ。
- 言語名は対象言語自身で表示する（`日本語`, `English`）。
- 設定変更のプレビューや確認は不要とし、選択後すぐに反映する。
- 言語選択と単位系選択を同一設定として混同しない。

### セットアップ、車両、比較

- 共有部品 `src/components/setup/SetupFormParts.tsx` のラベル、ヒント、必須表示、空状態から移行し、各タブの重複を減らす。
- フォーム項目の内部 ID、Firestore フィールド名、翻訳キーを分離する。
- 英語で長くなる見出しを想定し、固定幅・省略・折返しを実画面で確認する。
- `FL/FR/RL/RR`、製品名、型式など翻訳不要の表記を用語集で固定する。

### テレメトリ

- 分析結果の文章は、断片結合ではなくパラメーター付き翻訳文にする。
- グラフの凡例、軸ラベル、ツールチップ、単位、証拠バッジも翻訳対象に含める。
- トラック固有のコーナー名は、公式英語名がある場合のみロケール名を用意し、機械翻訳で上書きしない。

### 共有、画像、CSV

- 公開共有は閲覧者の端末言語を既定とし、`lang` クエリで固定できるようにする。
- 共有リンク作成時に言語を強制しない。送信者の言語が受信者へ恒久的に引き継がれないようにする。
- `shareImage.ts` の Canvas 描画文言、日付、フォント、文字幅をロケール対応する。
- CSV は `language` を明示して出力できるようにし、機械処理用の安定コード列と人向け表示列を混同しない。
- 日本語 CSV の既存利用者を壊さないため、既定出力は当面現在の日本語形式を維持する。

### HTML、PWA、SEO

- 実行時に `<html lang>` と title を更新する。
- `manifest.webmanifest` の多言語化はブラウザ対応差があるため、まず既定日本語を維持する。
- 現在の Vite SPA では共有ページごとの OGP と検索エンジン向け言語 URL を完全には出し分けられない。公開ページの SEO を重視する段階で SSR または Functions による HTML 生成と `hreflang` を検討する。
- 右から左へ書く言語を追加するときは `dir="rtl"`、アイコン方向、グラフ、数値入力を別途検証する。初期の日本語・英語では対象外とする。

## 6. 翻訳運用

### キーと文言のルール

- キーは削除や意味変更を避け、画面移動だけでは改名しない。
- 同じ日本語でも意味が違う語は別キーにする。
- `保存しました` のような汎用文を無理に共用せず、対象が明確なキーを使う。
- 翻訳文に HTML を入れない。リンクや強調が必要な場合だけ `Trans` を使う。
- aria-label、title、placeholder、alt、空状態、通知も翻訳漏れ検査の対象にする。
- 日本語を翻訳キーのフォールバックとしてコードへ残さない。フォールバックは日本語リソースで行う。

## 8. テスト方針

最低限、次を自動テストする。

- 言語決定の優先順と未対応ロケールのフォールバック
- 翻訳キーが日本語・英語の両リソースに存在すること
- 補間、複数形、0 件、1 件、複数件
- 日付境界とタイムゾーンを含む日付・時刻表示
- JPY、整数、小数、負数、null の表示
- 旧天候値と新天候コードが同じドメイン値へ正規化されること
- 新規保存では日本語列挙値が書き込まれないこと
- サービスのエラーコードが各言語のメッセージへ変換されること
- CSV の日本語版・英語版とエスケープ
- 公開共有 URL の `lang` 指定と不正値のフォールバック

実画面では、言語ごとに次の組合せを確認する。

| 対象 | 日本語 | 英語 |
| --- | --- | --- |
| デスクトップ / ライト | 必須 | 必須 |
| デスクトップ / ダーク | 必須 | 必須 |
| モバイル / ライト | 必須 | 必須 |
| モバイル / ダーク | 必須 | 必須 |
| 空状態・エラー状態 | 必須 | 必須 |
| 公開共有（未ログイン） | 必須 | 必須 |

確認項目は、情報密度、揃え、間隔、操作順、折返し、切詰め、モーダル高さ、グラフ凡例、フォーカス表示、aria 名とする。

## 9. 完了判定

多言語対応を「完了」と判断する条件は次のとおり。

- UI にユーザー向け日本語・英語文字列が直接記述されていない。
- 日本語・英語の翻訳キー集合が一致し、欠落時にキー文字列が画面へ露出しない。
- 保存される標準列挙値が言語非依存コードになっている。
- 自由入力・過去スナップショット・標準 UI 文言の区別が守られている。
- 日付・数値・通貨が選択中のロケールで表示される。
- Ant Design、Canvas、CSV、aria、HTML title、公開共有も対象に含まれている。
- 旧データの読取互換テストと主要導線の日本語・英語 E2E が通る。
- デスクトップ、モバイル、ライト、ダークの実画面レビューが完了している。

## 10. 実装開始時の注意

現在のワークツリーには多くの未コミット変更がある。多言語化は広範囲に触れるため、既存作業を先に整理し、Phase 0 を独立した小さな変更として開始する。最初の変更で全画面の文言を一括置換せず、データ互換と Provider を先に確立してから業務領域ごとに移行する。
