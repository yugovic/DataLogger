## 1. この文書の位置づけ

このリポジトリには、事業フェーズ、技術ロードマップ、コンテンツ差別化、WP仕様がそれぞれ存在する。一方で、状態と優先順位は複数文書に分散し、次の不整合がある。

- `BUSINESS_PLAN.md` の Phase 0a〜0c は未完了表示だが、`SPECIFICATION.md` では完了表示
- `docs/strategy/06-content-differentiation-plan.md` のWP一覧には、実装後の状態が反映されていない
- ランキング、カード交換、3Dリプレイは構想にあるが、同じ粒度のバックログ項目・リリース条件になっていない
- 3Dリプレイは別リポジトリで進行中だが、本体へ統合する契約と受け入れ条件が一元管理されていない

今後、優先順位、状態、リリース境界は本書を正本とする。詳細仕様は各文書に残し、本書から参照する。

状態は次の6種類に統一する。

| 状態 | 意味 |
|---|---|
| Released | ユーザーへ提供済みで、品質ゲートを通過している |
| Release candidate | 実装は存在するが、実利用・回帰・計測の確認が残る |
| Ready | 価値、スコープ、受け入れ条件が定まり着手可能 |
| Discovery | 仮説や体験を検証中。実装範囲はまだ固定しない |
| External trial | 別リポジトリ等で試作中。本体の完了とは数えない |
| Icebox | 意図的に着手しない。前提となる仮説が成立した場合のみ再評価する |

## 2. プロダクトの価値循環

```text
走る
  → 記録する
  → ランキングで現在地を知る
  → 愛車カードで人とつながる
  → ロガー分析で次の一手を決める
  → 3Dリプレイで追体験・比較する
  → データを共有・交換・購入する
  → 比較可能なデータが増え、体験価値が高まる
```

魅力品質は単独の装飾機能ではない。次の役割で事業へ接続する。

| 柱 | ユーザー価値 | 事業価値 | 最初に検証すること |
|---|---|---|---|
| タイムアタックランキング | 目標、承認、再挑戦 | 再訪、イベント獲得、データ密度 | 同条件ランキングが次回利用を生むか |
| 愛車カード／現場交換 | 自己表現、出会い、記憶 | 紹介、共有、供給者の関係グラフ | 交換後に相手の車両・走行を閲覧するか |
| データロガー分析 | 納得、成長、次の一手 | 継続、有料転換 | 分析後に行動が決まるか |
| 3Dリプレイ／ゴースト | 驚き、追体験、直感的理解 | SNS獲得、購入転換 | 2D分析より理解・共有・購入が増えるか |
| データ共有・取引 | 学習の近道、貢献、収益 | 手数料、ネットワーク効果、データの堀 | 検証済み実車データに支払い意思があるか |

## 3. 統合バックログ

### Stream A — 信頼できる記録・分析基盤

| ID | 項目 | 状態 | 次の完了条件 | 依存 |
|---|---|---|---|---|
| A-01 | セットアップ記録・比較・履歴 | Release candidate | 実機で主要導線、空状態、モバイル、ダークモードを確認 | なし |
| A-02 | ロガー取込・ラップ証憑 | Release candidate | 対応フォーマットごとの実データ回帰と誤証憑ゼロ | A-01 |
| A-03 | 自己ベスト／前回ロガー比較 | Release candidate | 1画面で「最大損失区間→理由→次の行動」が完結 | A-02 |
| A-04 | 間引きトレース永続化 | Ready | 3Dと他者比較に渡せる共通トレース契約を確定・保存 | A-02 |
| A-05 | 他者／購入データ比較 | Discovery | 匿名化、権限、条件差表示、購入前後の閲覧範囲を確定 | A-04, E-02 |

### Stream B — 愛車カードと現場のつながり

| ID | 項目 | 状態 | 次の完了条件 | 依存 |
|---|---|---|---|---|
| B-01 | VehicleProfile＋スペックカード | Release candidate | 車両、共有詳細、画像出力で内容と申告ラベルが一致 | A-01 |
| B-02 | カード共有画像・公開リンク | Release candidate | 非ログイン相手がカードから車両・実績を理解できる | B-01 |
| B-03 | カードビューア／保存アルバム | Ready | 他人のカードを保存し、一覧から再閲覧できる | B-01, B-02 |
| B-04 | 現場でのカード交換 MVP | Discovery | QRを読み、相互にカードを保存し、交換相手を後から辿れる | B-03 |
| B-05 | イベントデカール／参加記録 | Discovery | 主催者コードで当日の参加証をカードに付与できる | B-04, C-03 |
| B-06 | カードトレード | Icebox | 所有移転が本当に価値を生むと確認するまで実装しない | B-03 |

B-04のMVPでは「交換」を所有権取引にしない。現場で会った二人が、お互いの愛車カードを自分のアルバムへ保存する行為とする。これにより、投機性を避けつつ「誰と、どのイベントで会ったか」というコミュニティ価値を検証できる。

### Stream C — タイムアタックランキングとイベント

| ID | 項目 | 状態 | 次の完了条件 | 依存 |
|---|---|---|---|---|
| C-01 | 自己ベスト／車種別ランキング | Discovery | 証憑付きタイムのみで、サーキット×車種別順位を表示 | A-02, B-01 |
| C-02 | 近い仕様ランキング | Discovery | タイヤ区分等を開示し、比較条件に納得できる | C-01 |
| C-03 | 走行会当日ランキング | Discovery | イベントコード参加者だけで当日順位を表示 | C-01 |
| C-04 | 成長ランキング／自己更新 | Discovery | 絶対タイムが遅い層にも参加理由が生まれる | C-01 |
| C-05 | ランキングから比較へ | Discovery | 順位を見るだけで終わらず、許可された相手データとの比較へ進める | A-05, C-02 |

最初のランキングは総合順位だけにしない。初回リリースでは「同一サーキット×車種」「証憑付き」「自己ベスト更新」を中心にする。厳密な公平性を主張せず、カードでタイヤ・改造度等の文脈を確認できる設計にする。

### Stream D — 3Dリプレイ統合

| ID | 項目 | 状態 | 次の完了条件 | 依存 |
|---|---|---|---|---|
| D-01 | 3D単独リプレイ試作 | External trial | 別リポジトリで実データ1周を安定再生 | A-04相当の入力 |
| D-02 | 本体用データ契約 | Ready | 下記の統合契約を両リポジトリで固定 | A-04, D-01 |
| D-03 | 本体への読み取り専用統合 | Discovery | セッション詳細から遅延なく開き、失敗時も分析導線を壊さない | D-02 |
| D-04 | 共有リプレイ／短尺クリップ | Discovery | 公開範囲を守り、透かし・リンク付きで共有できる | D-03, B-02 |
| D-05 | 2台ゴースト比較 | Discovery | 同距離軸で2トレースを同期し、差が生じる区間を説明できる | D-03, A-05 |
| D-06 | 購入データのゴースト化 | Discovery | 購入権限と連動し、プレビュー／購入後の範囲を分離 | D-05, E-02 |

3D統合契約の最低要件:

- 入力はロガー固有形式ではなく、共通トレース形式にする
- 必須データ: `elapsedTime`、`distance`、相対座標または緯度経度、`speed`
- 任意データ: 標高、スロットル、ブレーキ、横G・縦G、ギア、RPM
- 単位、欠損、サンプリング間隔、開始／終了条件を明示する
- 位置情報の匿名化・公開範囲を本体側が決定できる
- 3Dモジュールは遅延読み込みし、本体の初期バンドルを増やさない
- WebGL非対応、ロード失敗、データ不足時は2D分析へ安全に戻る
- 単独再生だけを最終完成としない。D-05またはD-04への接続を価値検証の完了条件にする

## 4. 推奨リリース計画

日付ではなく、価値が成立したかをゲートにする。各リリースは一つの問いに答える。

### Release 0 — Trust Beta: 安心して記録し、振り返れるか

対象: A-01〜A-03、B-01、E-01

リリース範囲:

- セットアップ記録、履歴、比較
- ロガー取込、証憑付きラップ
- 自己ベスト／前回比較と次のアクション
- 愛車スペックカード
- Give-to-Get共有

リリース条件:

- 対応ロガーの実データで重大な誤読・偽証憑がない
- 初回ユーザーが説明なしで「記録→比較→次の行動」まで完了できる
- 公開／非公開と申告値／証憑値を誤認しない
- モバイル、空状態、エラー、ダークモードを実画面で確認
- 回帰チェックとビルドが通る

判定KPI:

- 初回走行記録完了率
- 2回目の記録率
- 分析後の「次のアクション」保存率
- 記録ユーザーの共有率

### Release 1 — Paddock Beta: 現場で人とつながり、また開きたくなるか

対象: B-02〜B-05、C-01、C-03、C-04

最小の一連体験:

```text
走行会コードで参加
  → 証憑付きタイムが当日ランキングへ反映
  → 現場でQRを読み合い、愛車カードを交換
  → 帰宅後、カードから相手の車両・公開走行を振り返る
```

リリース条件:

- 一つの実証走行会で10〜30人が利用可能
- QR交換が30秒以内で完了し、連絡先交換を強制しない
- ランキングは証憑付きで、車両条件をカードから確認できる
- 公開範囲とブロック／削除の扱いが明確
- 主催者の当日運用を増やしすぎない

判定KPI:

- 参加者のカード作成率
- 1人あたりカード交換数
- 交換翌日の再訪率
- ランキング閲覧率と次回走行時の再利用率
- 主催者の再導入意向

### Release 2 — Replay Alpha: データを見る体験が変わるか

対象: A-04、D-01〜D-04

最小の一連体験:

```text
ロガーを取り込む
  → 分析から3Dリプレイを開く
  → 差が大きい区間へジャンプする
  → 短いリプレイを共有する
```

リリース条件:

- 別リポジトリと共通トレース契約が固定されている
- 実データ複数形式・複数サーキットで再生できる
- 初期バンドルと通常の記録・分析体験を悪化させない
- 低性能端末／WebGL非対応時に安全なフォールバックがある
- 公開範囲外の位置・走行データを共有しない

判定KPI:

- リプレイ完走率
- 2D分析のみと比較した損失区間の理解率
- 共有率、共有リンクからの登録率
- リプレイ翌日の再訪率

### Release 3 — Ghost & Market Pilot: 他人の走りにお金を払うか

対象: A-05、D-05〜D-06、E-02〜E-05

最小の一連体験:

```text
商品カードとリプレイプレビューを見る
  → 条件と売り手の信頼を確認する
  → データを購入する
  → 自分の走りとゴースト比較する
  → 次走の改善点を持ち帰る
```

リリース条件:

- 先に手動販売でH3を検証する
- 商品にカード、証憑、条件、権利、説明がそろう
- 購入前プレビューと購入後データを権限で分離する
- 返金、削除、共有解除、再配布禁止の扱いが明確
- ゴースト比較が「面白い」だけでなく次の行動につながる

Go基準:

- データ購入率 5%以上
- リピート購入率 30%以上
- ゴースト付き商品の購入率が素データ商品より高い
- 購入者が次走でデータを再利用する

Go基準に届かない場合、決済機能を拡張せず、商品規格・対象マス・価格・プレビュー価値を見直す。

### Release 4 — Network Release: データ密度が価値を自走させるか

対象: C-02、C-05、E-06

リリース条件:

- 選定した5〜10の「サーキット×車種」マスに比較可能な在庫がある
- 共有・購入・ランキングが同じ条件軸で接続している
- 集合知が個別データを超える有料価値を生む

判定KPI:

- 重点マスごとの証憑付き在庫数
- 30日以内流動性 20%以上
- 有料転換率 2〜5%
- 比較可能な相手が見つかる検索率

## 5. 着手順の判断

現時点の推奨順は次の通り。

1. **Release 0を本当にリリース可能な品質へ閉じる**  
   実装済み表示を増やすより、実データ、実画面、計測で基盤の信頼を確定する。
2. **Release 1を小さな走行会で検証する**  
   ランキングとカード交換を別々に作らず、「当日会う→競う→交換する→後で振り返る」の一連で出す。
3. **並行して3Dの統合契約だけを先に固定する**  
   3D側の描画実装を待たず、共通トレース、権限、フォールバック、パフォーマンスの境界を決める。
4. **Release 2で単独リプレイと共有価値を検証する**  
   ここではまだ決済を必須にしない。
5. **手動販売実験の成立後にRelease 3へ進む**  
   Stripeを先に作らず、検証済みデータに支払い意思があることを確認する。

この順序により、カードとランキングがユーザー獲得・データ供給を作り、ロガー分析と3Dが継続・理解を作り、最後にゴースト付きデータ商品が収益へつながる。

## 6. 今は作らないもの

- 投機や希少性を主目的にしたカード所有権トレード
- 全車種・全サーキットを対象にした薄い総合ランキング
- 3D映像だけを独立して鑑賞する大規模機能
- 支払い意思の検証前の完全なマーケットプレイス決済基盤
- データ密度が不足した状態でのAI自動提案
- 比較条件を説明できない「公平ランキング」表示

## 7. バックログ更新ルール

各項目には最低限、次を持たせる。

- ユーザーのジョブ／解決したい場面
- 事業仮説と対象KPI
- 状態
- 依存関係
- 受け入れ条件
- 計測イベント
- 公開範囲とデータ品質上のリスク
- リリース後のGo／Iterate／Stop基準

運用ルール:

1. 新機能は必ず Stream と Release に所属させる
2. 仕様書ができても `Released` にはしない
3. コードが存在しても、実画面確認・回帰・計測がなければ `Release candidate` とする
4. 別リポジトリの成果は、本体で受け入れ条件を通るまで `External trial` とする
5. 状態変更時は本書を先に更新し、詳細文書は本書を参照する
6. 月次で重複項目、停止項目、KPI未計測項目を棚卸しする
