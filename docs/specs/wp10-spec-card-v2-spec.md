## ユーザー決定事項（2026-07-03）

1. **カードの主語は「車両」**（「ドライバー×車両」ではない）。ownerLabel は補助情報のまま維持
2. **デカール**: ユーザーが自分でカードをデコれる。獲得は記録由来、配置はユーザーの自由
3. **収集**: 方向性承認。今期は基盤（種まき）まで
4. **カードはモーダルで開く**（一覧のインライン展開を置換）
5. **3D回転**: ユーザーが自分の手でカードを回せる（three.js「とか」= 技術は任意、要件は操作感）

## 役割モデル（機能追加の振り分け原則）

カードは観る人により4役割（鏡=所有者 / 名刺=仲間 / 商品ラベル=データ閲覧者 / 広告塔=未登録者）を持つ。
**表面＝感情の面（誇れるもの）、裏面＝信頼の面（申告・前提条件）** — 以後の追加はこの二面に振り分ける。
レア度の軸は改造度カラー（既存）に加え、**お金で買えない第二軸（記録由来のデカール・証憑）**を導入する。

## プロジェクト原則（厳守）

- **null/偽データ原則**: デカールは記録データからの導出でのみ獲得（手動付与・購入・自己申告での作成は恒久的に不可）。
  戦績・デカールは該当データが無ければ要素ごと非表示（0埋め・ダミー禁止）
- **画面と共有画像の一致**: カード上の要素（デカール・戦績）は shareImage.ts の PNG 出力にも同一に描画する
- **スナップショット方針**: 公開スナップショットに車両写真は含めない（現行踏襲）。デカール配置は含める
  （獲得根拠を再検証したもののみ焼き込み）
- **低情報密度の原則**（944a349）: 表面の情報量を増やさない。デカールはヒーロー領域内のみ、
  ピル・車名・フッターの領域は不可侵
- **バンドル**: 新規の重い依存は追加しない。3DはまずCSS 3D transformで実装（追加依存ゼロ）。
  three.js/r3f はWP-3（3Dリプレイ、別プロジェクト合流待ち）と統合判断し、導入時も dynamic import 必須
- ダークモード対応・UIテキスト日本語・既存規約どおり

## スコープ

### 1. カードビューアモーダル＋3D回転 — `src/components/vehicle/CardViewerModal.tsx`（新規）

**導線（インライン展開の置換）**
- VehicleList: 「カード」アクション → モーダルを開く。既存の `expandedVehicleIds` によるインライン展開は廃止。
  「保存」アクションはモーダル内へ集約（一覧アクション行の混雑解消）
- SharedSetupDetail: カードのタップ/クリック → 閲覧専用モーダル
- PublicShareLanding: 同上（閲覧専用。保存・編集導線は出さない — 未登録者はCTAへ誘導）
- SetupCompare の compact 表示は現状維持（スコープ外）

**3D回転インタラクション**
- ドラッグ（マウス/タッチ）でY軸回転が指に追従。離すと近い面（表/裏）へスナップ
- ドラッグせずタップ = 従来どおりフリップ（既存操作の互換維持）
- 縦ドラッグで ±15° 程度の X軸チルト（「手に持っている」感）
- 回転角に連動して光沢（シーン）がスライドする CSS グラデーションオーバーレイを重ねる
  — 将来の「証憑箔」（役割モデル仮説F）の土台。角度は CSS カスタムプロパティで受け渡す
- 実装: `perspective` + `preserve-3d` + pointer events のCSS 3D。**three.js は使わない**
  （曲面・環境反射が要件化した時点でWP-3と合わせて再評価。要件はあくまで「自分の手で回せる」こと）
- キーボード: ←/→ で15°回転、Enter/Space でフリップ、Esc で閉じる
- `prefers-reduced-motion`: 追従回転・光沢を無効化し、ボタンによる即時フリップのみ
- モバイル: カード領域は `touch-action: none`（ページスクロールと干渉させない）。モーダルはほぼ全画面

**a11y（既存課題の解消を含む）**
- 非表示面に `aria-hidden` を付与（SpecCard 本体も修正 — 現状スクリーンリーダーが表裏を二重読み上げする）
- フォーカストラップ・Esc クローズ（AntD Modal の標準機能を利用）

**モーダル内アクション行**: 画像保存 / 共有 / デカール編集（所有者のみ）

### 2. デカールシステム — `src/lib/decals.ts`（新規・純関数） + 配置エディタ

実車がイベントステッカーやパドックパスを重ねていく文化のデジタル化。
**お金で買えず、記録でしか増えない**。改造度カラーと独立した第二のレア度軸。

**カタログ（静的定義、第1弾）**
```typescript
interface DecalDefinition {
  id: string;                       // 'circuit:suzuka-full' | 'milestone:first-log' 等
  kind: 'circuit' | 'milestone';
  label: string;                    // トレイ表示名（日本語）
  artId: string;                    // SVGアセット参照
}
```
- サーキットデカール: `tracks.ts` 収載サーキットごとに1種。セッション記録の circuit 文字列を
  正規化テーブル（トラックID＋別名）で解決。**未収載サーキットは第1弾では対象外**
- マイルストーンデカール（3種に絞る）: `first-log`（記録1件）/ `sessions-10`（累計10件）/
  `logger-first`（ロガー証憑付きセッション1件）
- イベントデカール（主催者コード配布）は将来・WP-9連動でスコープ外

**獲得判定（導出のみ・保存しない）**
- `deriveEarnedDecals(setups: CarSetup[]): Set<string>` — クライアントで都度導出。
  獲得集合を Firestore に保存しない（改ざん・実データとの乖離を構造的に防ぐ）
- 共有スナップショット生成時に再検証: 配置済みでも獲得根拠のないデカールは除外して焼き込む

**配置（所有者のみ、モーダル内「デカール編集」モード）**
- トレイ: 獲得済みデカール一覧＋未獲得はシルエット表示＋獲得条件文言（収集欲の演出）
- ヒーロー領域内へドラッグ配置。移動・回転・拡縮（0.6〜1.6倍）・削除。**上限8枚**
- 自動では貼らない（獲得しても未配置がデフォルト。「デコる」楽しみはユーザーの手に残す）
- 座標はヒーロー矩形基準の 0..1 正規化で保存（カード表示サイズに非依存）

**データモデル**
```typescript
interface CardDecalPlacement {
  decalId: string;
  x: number; y: number;        // 0..1（ヒーロー矩形基準）
  rotation: number;            // deg
  scale: number;               // 0.6..1.6
  z: number;                   // 重なり順
}
// VehicleProfile / PublicVehicleProfile に cardDecals?: CardDecalPlacement[] を追加
```
- zodスキーマ拡張（vehicleProfileSchema / 公開プロフィールスキーマ両方）。
  範囲外座標・過大scale・9枚以上は invalid。未知の decalId は**表示時にスキップ**（カタログ改廃への後方互換）

**描画**
- SVGアセット（インライン・1〜2色のラインアート基調でカードの品位を保つ）。写真ヒーローの上にも重なる
- shareImage.ts の Canvas 描画に同一配置で対応（画面との一致原則）
- compact 変種には表示しない

### 3. 収集基盤（種まき） — 保存コレクションと最小アルバム

トレカ形式が約束する「集める」の最小土台。走行会でのカード交換（将来・WP-9連動）を壊さない情報設計にする。

- **保存**: SharedSetupDetail（Give-to-Get圏内）のカード近くに「カードを保存」。保存時点のカードを複製として保持
  ```typescript
  // users/{uid}/collectedCards/{cardId}
  interface CollectedCard {
    savedAt: Timestamp;
    sourceShareId: string | null;   // 出所参照（ownerUid は保存しない — 匿名共有の秘匿を破らない）
    card: {
      carModel: string;
      profile: PublicVehicleProfile; // cardDecals・（C-4導入後は）戦績を含む
      ownerLabel: string | null;     // 共有側の匿名化設定に従った値をそのまま複製
      capturedAt: Timestamp;
    };
  }
  ```
- **削除セマンティクス**: 画像保存と同等の扱い（元共有が解除されても保存カードは残る）。
  出所は shareId 文字列のみ保持し、逆引きで所有者を特定できる情報は持たない
- **最小アルバム** `/cards`: 集めたカードのグリッド表示 → タップでモーダル3Dビュー（閲覧専用）
- **Firestoreルール**: `users/{userId}/collectedCards` の own read/create/delete を追加。
  ※ルールは publicShares 分も含め**未デプロイ**（運用ブロッカー）。本WPのデプロイ時に一括で
  `firebase deploy --only firestore:rules,firestore:indexes` を実施すること
- スコープ外: QR交換・「今日のパドック」アルバム・トレード・公開LPからの保存

### 4. 戦績刻印【着手保留 — 未決事項の決定後に開始】

役割モデルから導いた「表＝代表ベスト1本、裏＝サーキット別戦績表」（モックA＋D構成）。

- 表面: 車名下の余白に刻印行 — BEST LAP・サーキット名・タイム・ロガー証憑バッジ（証憑がある場合のみ）
- 裏面: サーキット別ベスト一覧（証憑ありのタイムにのみバッジ）
- データ導出: 所有者ビューは自分の setups（vehicleId紐付き）から `lapTimeToMs`（`src/lib/setupFields.ts`）で
  最速を導出。共有スナップショットは生成時に焼き込み
- **共有面での戦績表示は車両ごとに opt-in**（ワンメイク参戦者の競技機密性への配慮）
- 記録が無ければ行ごと非表示（null原則）
- **未決**: 表面ヘッドラインの形式 — ①絶対ベスト / ②成長差分（例: 今季 −2.3s、ビギナーの感情ジョブ向き）/
  ③所有者が選択式。決定後に着手する

## スコープ外（やらないこと）

- three.js / React Three Fiber の導入（WP-3 合流時に再評価。C-1 は CSS 3D で成立させる）
- 購入可能デカール・手動作成デカール（偽データ原則違反のため恒久的に不可）
- QR交換・走行会アルバム・カードトレード（収集の将来拡張）
- イベントデカール（WP-9 主催者コードと連動するため）
- 証憑「箔」ホロ演出の本実装（C-1 の角度連動光沢を土台に将来検討）
- 車両写真の公開スナップショット掲載（opt-in公開の議論は別途）
- SetupCompare の表示変更

## 受け入れ基準

- [ ] `npm run typecheck` / `npm run lint` / `npm run test` 全成功。既存テストに回帰がない
- [ ] 一覧・共有詳細・公開LPからカードがモーダルで開き、ドラッグで回転→離すと表裏いずれかにスナップする

- [ ] cardDecals の不正値（範囲外座標・9枚以上・過大scale）が zod で invalid になる（unit: schema）
- [ ] 保存した他人のカードが /cards に表示され、元共有の解除後も閲覧できる
- [ ] collectedCards に所有者を逆引きできる情報（ownerUid等）が含まれない
- [ ] 初期バンドルサイズが増えない（新規依存なし。`npm run build` のチャンクサイズで確認）
