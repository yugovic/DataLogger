# Codexサブエージェントによる企画・実装・レビューワークフロー

このドキュメントは、Codexのサブエージェント機能を使用して、企画→実装→レビューのプロセスを自動化する方法を説明します。

## 前提条件

以下のCustom Agentが `~/.codex/agents/` に定義されている必要があります：
- `planner.toml` - 企画・設計エージェント
- `implementer.toml` - 実装エージェント
- `reviewer.toml` - レビューエージェント

## パターンA: Custom Agentを個別に使用

### フェーズ1: 企画（plannerエージェント）

```bash
codex "plannerエージェントとしてユーザー管理ダッシュボードの企画・設計を行ってください"
```

plannerが以下を作成：
- 要件定義
- 画面設計
- データモデル
- API仕様
- 技術スタック選定

### フェーズ2: 実装（implementerエージェント）

```bash
codex "implementerエージェントとしてSPECIFICATION.mdに基づいて実装を行ってください"
```

implementerが以下を実行：
- 企画ドキュメントの読み込み
- TODOリスト作成とタスク分割
- 並列処理での実装
- 依存関係の管理

### フェーズ3: レビュー（reviewerエージェント）

```bash
codex "reviewerエージェントとして実装コードを企画書と照合してレビューしてください"
```

reviewerが以下を実行：
- 実装コードの読み込み
- 企画ドキュメントとの照合
- 品質評価（可読性、保守性、パフォーマンス、セキュリティ）
- 改善点とバグの特定
- レビューレポート作成

## パターンB: /fleetコマンドによる並列実行

GitHub Copilot CLIを使用している場合：

```bash
/fleet ユーザー管理ダッシュボードの開発:
1. 企画: 要件定義、画面設計、データモデル、API仕様を作成（plannerエージェント）
2. 実装: 企画書に基づいてコードを実装（implementerエージェント）
3. レビュー: 実装コードを企画書と照合してレビュー（reviewerエージェント）

依存関係: 実装は企画完了後、レビューは実装完了後
```


### フェーズ2: 実装

### フェーズ3: レビュー

## サブエージェントの管理

### アクティブなエージェントの確認

```bash
/agent
```

### エージェントの停止

```bash
codex "実行中のエージェントを停止してください"
```

### エージェントスレッドのクローズ

```bash
codex "完了したエージェントスレッドをクローズしてください"
```

## メリット

- **モデルの最適化**: 各タスクに最適なモデルと設定を使用
- **コンテキスト分離**: 各エージェントが専用のコンテキストで動作
- **並列処理**: /fleetコマンドで複数のエージェントを並列実行可能
- **自動オーケストレーション**: Codexがサブエージェントの生成と調整を自動処理
- **再利用性**: Custom Agent定義を再利用可能

## 設定オプション

### グローバル設定（~/.codex/config.toml）

```toml
[agents]
max_threads = 6           # 最大並列スレッド数
max_depth = 1            # 最大ネスト深さ（デフォルト: 1）
job_max_runtime_seconds = 1800  # ジョブの最大実行時間
```

### Custom Agentの追加設定

各Custom Agentファイル（.toml）で以下の設定が可能：

```toml
model = "gpt-4o"                    # 使用するモデル
model_reasoning_effort = "medium"   # 推論 effort
sandbox_mode = "read-only"          # サンドボックスモード
mcp_servers = ["slack"]            # MCPサーバー
skills.config = {...}              # スキル設定
```

## 注意点

- サブエージェントは明示的に依頼した場合のみ生成されます
- サブエージェントワークフローは単一エージェント実行より多くのトークンを消費します
- `max_depth` をデフォルトの1以上に設定すると、再帰的な委譲が可能になりますが、トークン使用量とレイテンシーが増加します
- 非対話モードでは、新しい承認が必要なアクションは失敗します
