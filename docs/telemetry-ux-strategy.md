# データロガー表示のUX戦略 — 顧客ニーズ分析と体験最適化

作成: 2026-06-13 / 企画・分析: Fable / ステータス: 検討用ドラフト（実装前の合意形成資料）

> 本書は実装仕様ではなく、「データロガーをどう見せ、走行ログとどう紐付け、どう比較させるか」を
> 顧客ニーズ起点で再定義する戦略資料。BUSINESS_PLAN.md のデータ中心戦略・3つの体験
> （楽しむ / 高みを目指す / 成長する）に接続する。

---

## 0. 結論（先に要点）

1. **データロガーは「機能」ではなく「成長ループの証拠層」である。** 単独の分析ページとして持つのをやめ、
   「セッション（=セット+条件+テレメトリ+ラップ+感想）」を中心オブジェクトに据え、記録と分析を一体化する。
2. **表示の主役を「グラフ」から「差分（デルタ）」へ移す。** ユーザーが知りたいのは数値ではなく
   「どこで・なぜ・何秒失ったか」「セットを変えて効いたか」。最重要チャートは現状欠けている
   **累積タイム差（デルタT）トレース**。
3. **比較は4軸で設計する。** ①周回内 ②セッション間（前回vs今回）③自己ベスト ④マーケットの検証済みデータ。
   ②③④を成立させる鍵が「**間引きトレースの永続化**」という一つの判断。
4. **セット変更とテレメトリ変化を横並びで見せる＝因果の可視化が我々の堀。** テレメトリ表示の美しさでは
   競合（Garmin Catalyst, AIM等）と戦わない。「セット↔テレメトリ↔タイム」の紐付けで差別化する。
5. **動画連携は「タイムライン同期」を核に段階導入し、最終的にマーケットの最上位商品（オーバーレイ付きオンボード）にする。**
6. **生テレメトリを「読める人」は実質コーチ／データ屋に限られる。解釈支援レイヤー（自動アノテーション＋平易な読み解き＋セット↔ドライバーの切り分け）を上にかぶせ、専門知識のギャップを埋める。** これは在庫（記録）を価値に変える活性化・継続の鍵であり、同時に上級者が「解説つきデータ」を売る新商品も生む。

## 1. 現状の表示の課題（実装の率直な評価）

`/telemetry`（TelemetryAnalysis.tsx）は単体としては良くできている（ラップ一覧→2本選択→速度/前後G vs 距離 + 走行ライン + タイム差）。
しかし事業の狙いに対して構造的な弱点がある:

| # | 課題 | 何が問題か |
|---|---|---|
| C1 | **揮発的** | 訪問のたびにファイル再取込。記録が残らず、縦断的（時系列）価値がゼロ。「先月と比べる」ができない |
| C2 | **サイロ化** | `/telemetry` がセットアップ記録と切れている。「このベストラップを出したセットは何か」が辿れない（証憑バッジは記録→テレメトリの一方向のみ） |
| C3 | **比較が周回内に限定** | 同一ファイル内の2周しか比べられない。前回セッション・自己ベスト・他人のデータと重ねられない |
| C4 | **「で、どうすれば?」が弱い** | 速度/Gは出るが、**累積タイム差トレース**（どこで時間を得失したかを積算で見せる、プロツール必須の図）が無い。判断に直結しない |
| C5 | **セット文脈が不在** | テレメトリの隣に「どんなセットだったか」が出ない。因果（セット→結果）を読み取れない |

→ 要するに、いまは「ログビューア」。目指すべきは「**変更→計測→学習→再変更の成長ループを回す道具**」。

## 2. 顧客ニーズ分析（Jobs to be Done）

セグメント別に「片付けたい用事」で捉える。テレメトリ表示が応えるべきは主にこれら:

### 伸び悩む中級者（コア買い手・約3〜5万人）
- **JTBD-1「今日の走り、どこで失った?」** — 周回内のセクター/区間ロス。➡ デルタTトレース・セクター差
- **JTBD-2「セット変えたけど、効いた?」** — 前回との差。**セット差分とテレメトリ差分とタイム差を同時に**見たい。
  ➡ これが最も深い未充足ニーズ。実車・日本語でこれを提供する競合は不在
- **JTBD-3「速い人は何が違う?」** — 検証済みデータとの重ね。➡ マーケットのリファレンス・ゴースト
- 支払い意思の源泉はここ。走行会1回（2〜3万円）の試行錯誤コストを、データで短縮できるなら数千円は安い

### 記録習慣のないビギナー（裾野・流入源）
- **JTBD-4「成長してる実感がほしい」** — 上達の可視化。➡ ベストラップの時系列、「自己ベスト更新!」の瞬間演出
- 注意: テレメトリのグラフは**怖い・読めない**。彼らにはまず「タイム」と「前回より速い/遅い」で十分。
  グラフは見たい人だけが掘る設計（プログレッシブ・ディスクロージャ）

### 上級者・セミプロ / ショップ（供給の核・売り手）
- **JTBD-5「知見を信頼できる形で見せたい・売りたい」** — 証憑つきで権威的に。➡ テレメトリ表示＝商品パッケージそのもの
- 売り手の実績ページがポートフォリオになる。テレメトリ+セット+（将来）動画のバンドルが商品単位

## 3. 中核となるUXの再定義

### 3.1 「セッション」を中心オブジェクトにする
テレメトリを独立機能にせず、すべてを **セッション** に束ねる:

```
セッション = セット（静的設定）＋ コンディション ＋ テレメトリ（動的記録）
            ＋ ラップタイム（結果）＋ ドライバー感想（主観）＋〔将来〕動画
```

- セットアップ記録から「このセッションのテレメトリを見る」、テレメトリから「このラップのセットを見る」を双方向に
- 記録＝分析の入口。別ページに行かせない

### 3.2 表示を「デルタ・ファースト」にする
すべての図を「どこで・なぜ・どれだけ差がついたか」へ収束させる。
- **デルタT（累積タイム差）トレース**を最上段に追加 ＝ 現状最大の欠落。「S2入口から立ち上がりで0.3秒失っている」が一目で分かる
- 速度/G/ラインは「なぜそこで失ったか」の裏取りとして従属配置

### 3.3 スキルに応じた段階開示
同じデータを3層で。ビギナーを置き去りにせず、上級者を退屈させない:
1. **要約層**: ベストラップ・前回比（速い/遅い）・自己ベスト更新バッジ
2. **区間層**: セクター/区間別のデルタ、最高速
3. **詳細層**: 全トレース・走行ライン・生データ（掘った人だけ）

## 4. 比較機能の充実 — 4つの比較軸と「因果の可視化」

### 4.1 比較の4軸
| 軸 | 比較対象 | 応えるJTBD | 必要な土台 |
|---|---|---|---|
| ① 周回内 | 同一走行の2周 | JTBD-1 | 実装済み |
| ② セッション間 | 前回 vs 今回（同じ車×コース） | **JTBD-2** | トレース永続化（§5） |
| ③ 自己ベスト | 今 vs 自分の最速 | JTBD-1/4 | トレース永続化 |
| ④ マーケット | 自分 vs 検証済み他人データ | **JTBD-3** | 永続化＋共有/購入 |

### 4.2 因果の可視化（＝堀の中核）
②④の比較では、テレメトリ差分の隣に**セット差分**を必ず並べる:

```
［セット差分］                    ［結果］
フロントキャンバー −2.5° → −3.0°   ラップ −0.4秒
フロント空気圧    190 → 185 kPa     S2中間速度 +3 km/h
                                    デルタT: S2立ち上がりで挽回
```

これが「何を変えれば速くなるか分からない」というコアニーズへの直接の回答であり、
Excel・既存ロガーアプリ・海外勢が誰も実車×日本語で提供していない価値。**テレメトリ表示の最終目的はこの一画面**。

### 4.3 コンディションの正直さ
気温・路温が違う比較は林檎と蜜柑。条件を必ず併記し、差が大きい時は注意を出す。
将来は「条件の近い他人データを提示」（BUSINESS_PLAN Phase 2 の条件マッチング）へ。

### 4.4 操作チャンネルの重ね合わせと同期カーソル（「操作の違い」を見る）
速度・タイムは「結果」。その手前の「**操作の違い（原因）**」＝スロットル/ブレーキ/舵角/ギアを重ねて初めて、
なぜ差がついたかが分かる。
- **複数チャンネルを距離軸で同期したカーソルで串刺し** — デルタT・速度・操作・走行ライン（＋将来は動画）を
  1本のカーソルで貫き、「ある地点で、速度がいくつ違い、ブレーキ/アクセルがどう違ったか」を同時に読む
- **show / hide / overlay をユーザーが選ぶ** — 必要な軸だけ出す。常に全部出すと情報過多で読めなくなる。
  「速度だけ」「速度＋ブレーキ」「全部」を切り替えられること自体がUXの肝
- **デルタT線は瞬間ゲイン/ロスで色分け**（参照UIのように、その区間でどちらが速いかを赤/緑で）。
  「線がどっちに動いたか」で得失が直感的に分かる
- これは「比較機能の充実」の中核であり、§5 の永続化が効く（操作チャンネルも間引きで保存対象に含める）

### 4.5 解釈支援 — 専門知識のギャップを埋める（活性化の最重要論点）
**課題**: §4.2〜4.4 の表示を作っても、それを読み解けるのは実質コーチ／データエンジニア級。
コア顧客の中級者・ビギナーには重すぎる。**「見せる」だけでは価値が出ない。「読める」ようにして初めて在庫が価値になる。**

**解 = 生トレースの上に「解釈支援レイヤー」をかぶせる**:
1. **自動アノテーション** — 「T1でブレーキ15m早い」「立ち上がりでスロットル20%遅い」「中コーナーでアクセル抜き」等を
   チャンネル差からアルゴリズム検出し、デルタT上の発生地点にタグ表示。生トレースを「出来事の物語」に翻訳する
2. **平易な読み解き（コーチの声）** — 「S1で0.2秒損失、主因はT1の早すぎるブレーキ。S2で0.5秒挽回、立ち上がりのアクセルが良い」
3. **伸びしろの優先順位** — 最も改善余地の大きい1箇所を提示（「ここだけで0.2秒」）。全部を均等に見せない
4. **セット↔ドライバーの切り分け** — 差が「グリップ/ライン（=セットの差）」由来か「操作（=ドライバーの差）」由来かを分解。
   中級者の「セットのせい? 自分のせい?」という最大の問いに答える。これは判断支援（WP7 の目標空気圧）の自然な拡張
5. **段階的な深さ** — 一行の結論 → コーナー別 → 生トレース。読みたい人だけが掘る（§3.3 の段階開示）

**実装の順序とAIの位置づけ**: まず**ルールベース**（決定的・信頼できる・安価）で自動アノテーションと読み解きを作り、
後段で**AIによる自然言語化**を重ねる。重要: 旧「AIアドバイス」タブは保存に未接続の偽UIだったため WP7 で廃止したが、
いまは**ラップ検出済みの実テレメトリという土台がある**ため、データに根ざした読み解きとして正当に再導入できる。
ただし BUSINESS_PLAN の通り AI は堀ではない＝**活性化UXとして扱い、堀はあくまでデータ資産とマーケット**に置く。

**もう一つの機会 — 人の解釈を商品化する**: 解釈ギャップは上級者／コーチにとって商機でもある。
彼らは自分のラップに注釈（「100m看板でブレーキ」「ここは2速で」）を付け、「**解説つきデータ**」として売れる。
生データより明確に高付加価値で、マーケットの商品階層を一段引き上げる（§6.3 の動画と並ぶプレミアム軸）。

### 4.6 目標とする比較画面（参照レイアウト）
比較画面の到達イメージ = 速度オーバーレイ＋チャンネル切替＋セクター色分けコースマップ（現在位置ドット・セクターデルタ）
＋指標デルタカード＋セクター比較表＋同期オンボード映像、を一画面に統合したダッシュボード。

**優れている点**:
- チャンネル切替（速度/スロットル/ブレーキ/ステアリング/Gフォース）= §4.4 の操作チャンネルそのもの
- セクター色分けコースマップ＋位置ドット = 「コース上のどこか」を空間的に固定。グラフの距離軸と対応づく
- 指標デルタカード（最高速 +3.5km/h、ブレーキングポイント +14m、最小コーナリング速度 +4.5、フルスロットル時間、平均G）
  = 生トレースを"比べられる数値"に蒸留した**軽い解釈**。非専門家でも「14m手前でブレーキ」は直感的に分かる（§4.5 の入口）
- 同期オンボード映像（並置）= §6 動画の実用的な第一歩

**重要な注記**: このレイアウトは WP5 で廃止した `TelemetryComparison` デモの設計とほぼ同じ。
**廃止理由はデザインではなくハードコードの偽データ**だった。よって"設計はほぼ手元にある"。
やるべきは、この画面を**実パース済みテレメトリ＋永続化トレース（§5）で駆動する**こと。デモの再利用＝偽データ復活には絶対にしない。

**この画面の上に重ねる3層（＝我々の差別化）**:
1. **デルタTトレース（§4.2）** — 速度オーバーレイとセクターデルタはあるが、連続の累積タイム差が無い。
   「どこで得失したか」を最も直接に示すので、チャンネルか主ビューとして追加する
2. **コーチの読み解き（§4.5）** — 指標カードは"何が違うか"まで。"だから何をすべきか"（最も伸ばせる1点）を平易な言葉で上に乗せる
3. **セット差分ストリップ（§4.2 因果）** — これはテレメトリ単独の画面。どのセットがラップA/Bを生んだかを併記して初めて堀になる

**密度の注意**: 6チャンネル＋マップ＋表＋映像は情報過多になりやすい。上級者・売り手には最適だが、
中級者・ビギナーには §3.3 の段階開示（簡易/詳細トグル）で入口を用意する。
**相互作用の肝**: 速度グラフ・コースマップのドット・映像の再生位置を**同期カーソルで連動**させること（静的パネルの並置では価値が半減）。

## 5. これを可能にする一つの判断 — トレースの永続化

現状、生テレメトリは端末内処理のみ（Firestore 1MB制限）。これが C1・C3 と比較軸②③④を塞いでいる。

**判断: ラップ単位の「間引きトレース」を永続化する。**
- 速度/スロットル/ブレーキ/舵角を距離5〜10m間隔（1周あたり数百点）に間引けば数十KB。1MB制限内に十分収まる
- 比較②③④（どこで得失したか）にはこの解像度で必要十分
- 生の高頻度データは深掘り用に再取込（ローカル）に残す＝二層構造
- 証憑としての信頼は維持（間引きは表示用、ラップタイム算出は生データから）

> これは実装詳細だが、**何ができる体験かを規定する分岐**なので戦略として明記する。
> 永続化しない限り「先月と比べる」「速い人と比べる」は永遠に作れない。

## 6. 動画連携の構想（将来）

### 6.1 核心は「タイムライン同期」
テレメトリの時間軸と動画のタイムスタンプを結ぶ。片方をスクラブすると、もう片方が追従する。
- **オーバーレイHUD**: 速度・回転数・G・ミニ走行ライン（現在位置ドット）・デルタT を映像に重ねる
- 同期手段: 手動（既知イベントで合わせる）または GPS/音声による自動同期
- 映像ソース: オンボード（GoPro・スマホ）/ コーナーで撮った外部映像

### 6.2 3つの体験への効き方
- **楽しむ**: 走りを追体験・ハイライト共有（SNS拡散＝CAC低減にも効く）
- **高みを目指す**: 速い人のブレーキング点・ライン取りを「映像＋数値」で学ぶ
- **成長する**: 映像の成長日記（半年前の自分との比較）

### 6.3 マーケット商品としての進化
商品単位を段階的に高付加価値化:
```
検証済みセット → ＋テレメトリ → ＋オーバーレイ付きオンボード動画
（¥500〜）      （¥1,000〜3,000） （プレミアムSKU）
```
「速い人の決定的ラップを、データを重ねた映像で買える」は強力な差別化かつ高単価商品。

### 6.4 留意（先に設計フックだけ持つ）
- プライバシー: 顔・ナンバー・場所が映る。既存の `anonymized` フラグと連動させる思想を今から持つ
- ストレージ費用: 動画は重い。Phase 2 以降、需要実証後。ベータでは作らない

## 7. 段階的ロードマップ（事業フェーズ連動）

| 段階 | 内容 | 狙うJTBD | コスト感 |
|---|---|---|---|
| **A（直近・低コスト高効果）** | セッション紐付け（双方向導線）＋ デルタTトレース ＋ 操作チャンネル重ね・同期カーソル・show/hide ＋ セット文脈の併記 ＋ ルールベースの自動アノテーション（基本） | 1, 5 | 小〜中 |
| **B（リテンションの核）** | 間引きトレース永続化（操作チャンネル含む）→ セッション間・自己ベスト比較。セット差分↔テレメトリ差分の因果ビュー ＋ コーチ読み解き・セット↔ドライバー切り分け | **2**, 4 | 中 |
| **C（収益化）** | マーケットのリファレンス重ね（購入ラップをゴースト）＋ 条件マッチング ＋ 解説つきデータの出品（人の注釈を商品化） | **3** | 中 |
| **D（将来）** | 動画タイムライン同期＋オーバーレイ → 動画をプレミアム商品化 | 楽しむ/学ぶ | 大 |

※ 解釈支援（§4.5）は段階Aでルールベースの種を蒔き、Bで読み解き・切り分けへ、Cで人の注釈の商品化へと育てる横断テーマ。

段階Aは既存資産（パーサー・ラップ検出・echarts）でほぼ実現可能。Bが体験価値とリテンションの本丸。

## 8. 競合と差別化の整理

Garmin Catalyst / AIM / Harry's LapTimer / RaceRender / Track Titan は、テレメトリ＋（一部）動画を既に提供。
**テレメトリビューアそのものはコモディティ化している。** よって:

- ❌ グラフの綺麗さ・チャート種類の多さで戦わない（誰でも作れる＝堀にならない）
- ✅ **「セット↔テレメトリ↔タイムの因果の紐付け」と「検証済みリファレンスのマーケット」**に表示投資を集中する
