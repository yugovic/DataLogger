## 0. 深掘り: この要望の真因

表面的な要望は「3Dリプレイ」「チューニングカーのカード比較」だが、動詞化して深掘りすると:

1. 「3Dで軌跡を見たい」→ **自分の走りを"体験として"追体験・共有したい**
   - なぜ? 数値やグラフは分析には有効だが、感情が動かない。走った記憶と結びつかない
   - なぜ? 感情が動かないコンテンツは、SNSで共有されず、翌日に開かれない（NSM=習慣化に直結）
2. 「チューニング仕様をカードで比較したい」→ **タイム差の"言い訳と納得"を可視化したい**
   - なぜ? ワンメイク以外では素のタイム比較が不公平で、比較体験が「悔しいだけ」で終わる
   - なぜ? 「あの車はSタイヤで300psだから」という文脈がないと、比較から学びも購買動機も生まれない
3. 到達した真因（否定形の欲求）:
   - **「自分の走行が数字の羅列で終わってほしくない」**（記録が資産にも思い出にもならない不安）
   - **「不公平な比較で自信とモチベーションを削られたくない」**

つまりこの要望の本質は「分析機能の追加」ではなく、**記録データを『誇れるコンテンツ』と『公平な文脈』に変換すること**である。これは事業計画のデータマーケットプレイス構想と完全に整合する: 誇れるコンテンツ＝共有・出品動機（供給）、公平な文脈＝購買判断材料（需要）。

## 1. ペルソナ別JTBD分析

### P1: ワンメイク参戦者（例: ロードスター・パーティレース、86/BRZ杯、ヴィッツレース）
- **機能的ジョブ**: レギュレーション内の微差（内圧0.1、減衰1クリック）で予選0.3秒を削る。練習走行→予選→決勝のPDCAを週末単位で回す
- **感情的ジョブ**: 「マシンは同じ。腕の差だ」という世界で、負けの原因を特定して次に希望を持ちたい
- **社会的ジョブ**: 同クラスのライバルとの序列。パドックで一目置かれたい
- **決定的インサイト**: ワンメイクこそ**タイム比較が最も公平＝比較コンテンツの価値が最大**。しかしライバルはデータを出したがらない → Give-to-Getと匿名化が効く。また「同型車のデータ」は購入価値が最も明確（H3検証の最有力セグメント）
- **未充足ニーズ**: レースウィークエンドモード（練習→予選→決勝の時系列デブリーフ）、クラス内デルタ比較、レギュレーション適合チェック済みデータの信頼マーク

### P2: チューニングカー・タイムアタッカー（筑波アタック勢、峠→サーキット転向組）
- **機能的ジョブ**: 「次の1秒」に最も効く投資（改造 or 練習 or タイヤ）を見極める。シーズンをまたぐ進化の記録
- **感情的ジョブ**: 自分だけの一台を育てる物語の主人公でいたい。ビルドの歴史が誇り
- **社会的ジョブ**: 「あの仕様であのタイムはすごい」と正当に評価されたい。パワーだけの車と一緒にされたくない
- **決定的インサイト**: このセグメントの車は**一台一台が作品**。マシンスペックカードは比較の道具である以前に**自己表現メディア**。Attack系イベントのクラス分け文化（タイヤ・駆動方式・排気量区分）が既に存在し、自動クラス判定の下地がある
- **未充足ニーズ**: 改造履歴とタイム推移の紐付け（Mod ROI: この投資で何秒買えたか）、仕様正規化されたクラス別ランキング、ビルドストーリーの発信手段

### P3: 走行会エンジョイ勢・ビギナー
- **機能的ジョブ**: 安全に、確実に、前回より上手くなる。何を変えればいいか誰かに教えてほしい
- **感情的ジョブ**: 初サーキットの高揚を形に残したい。怖さと隣り合わせの挑戦を認められたい
- **社会的ジョブ**: 仲間に「走ってきた」と誇れる成果物がほしい（現状はスマホの流し撮り写真のみ）
- **決定的インサイト**: このセグメントは分析しない。しかし**「共有できる成果物」への渇望が最も強い**。3Dリプレイ動画やラップサマリー画像（shareImage.ts の延長）が刺さるのはここ。かつ最大人口セグメント＝獲得エンジンの主役
- **未充足ニーズ**: 自動生成される「今日の走行」ハイライト、専門用語なしの読み解き、成長の可視化（バッジ・自己ベスト履歴）

### P4: アンカー供給者（上級者・セミプロ・ショップ）
- **機能的ジョブ**: 知見を収益化・集客に変える。顧客の車両を継続管理する
- **感情的ジョブ**: 長年の経験知が正当な対価を得る場がほしい
- **社会的ジョブ**: 「このショップのデータは信頼できる」というブランド
- **決定的インサイト**: 出品物の見た目が価値を決める。**3Dゴースト付き・解説付き・スペックカード付きのデータ商品**は、CSVの切り売りと比べ体感価値が桁違い → H3の購入率5%達成の鍵
- **未充足ニーズ**: ポートフォリオページ、ブランド化されたカード、顧客車両フリート管理

### P5: 走行会主催者（GTMチャネル）
- **機能的ジョブ**: イベントの付加価値向上・リピーター獲得
- **決定的インサイト**: 「参加者全員に走行リプレイと比較データが残る走行会」は主催者の差別化商材になる。イベント単位のリーダーボード＋当日デブリーフはCAC≈0の獲得装置

## 2. 差別化の柱（3本柱＋横串）

競合分析（docs/strategy/01）の白空地「セットアップ記録×テレメトリ×データ売買」の上に、**コンテンツ資産化**という第4の軸を立てる。Track Titan/MyRaceLabは「分析ツール」であり、走行データを「誇れるコンテンツ」に変換する競合は世界に存在しない。

### 柱A: シネマティック・リプレイ（Three.js 3D軌跡再生）【保留 — 別プロジェクトで実施中】

> 2026-07-02 ユーザー判断: 3D化は別プロジェクトで進行中のため本プランでは保留。構想は将来の合流のため残す。共有コンテンツ（柱の横串）は静止画ベース（shareImage.ts拡張）で先行する。

数値分析ではなく**体験の再生装置**として設計する。分析はComparisonCockpit（実装済み）が担う。役割分担を明確に。

| 段階 | 内容 | 価値 |
|---|---|---|
| A-1 | 3Dコースリボン + 走行ライン + 再生カーソル。速度/デルタでリボンを着色。ブレーキ点マーカー | 「自分の走りが見える」基礎体験 |
| A-2 | **ゴーストバトル再生**: 2ラップ同時再生（自己ベスト/前回/共有データ/購入データ）。追い抜かれる瞬間・離される区間が直感で分かる | 比較コックピットの感情版。購入データの「試乗」体験 |
| A-3 | **共有クリップ自動生成**: ベストラップの3Dリプレイを15〜30秒の動画/GIF＋サマリーカードで書き出し（ロゴ透かし入り） | SNS拡散→獲得エンジン。P3の最重要成果物 |
| A-4 | カメラ演出（チェイス/俯瞰/TV風）、実写動画同期（既存段階D構想と合流） | プレミアム化・サブスク訴求 |

- **技術要点**: GPS由来のx/y相対座標・速度・時刻は取得済み。標高データはないため、まず**様式化された2.5D表現**（フラットなコースリボン＋速度で高さ/色をマッピングする抽象表現）で開始し、実地形（国土地理院標高タイル等）は後続。react-three-fiber をルート単位で dynamic import（バンドル2.8MB問題に配慮）
- **差別化の本質**: 「ゴーストと走る」体験をマーケットプレイス商品に接続する（購入したプロのラップと3D空間で追いかけっこ）。**データ商品を"読む物"から"対戦相手"に変える**。これはH3の支払い意思を根本から引き上げる仕掛けであり、シムレーシングで実証済みの心理（Coach Daveのゴースト文化）を実車に輸入する世界初の試み

### 柱B: マシンスペックカード（比較の公平性＋自己表現メディア）

トレーディングカード風のビジュアルカードで車両仕様を表現し、**すべてのタイム・セットアップ・テレメトリに文脈を添付**する。

- **カード構成要素（改造申告リスト中心）**: 車種/年式、駆動方式、タイヤ（銘柄・サイズ・区分）、**改造パーツ申告リスト**（カテゴリ別: 吸排気/過給/足回り/ブレーキ/エアロ/軽量化/ECU/駆動系…、パーツ名・メーカー名は自由記述）、ベストタイム（サーキット別・証憑付き）、オーナー/ショップ名（匿名化可）。パワー・車重は**任意の自己申告**（大半のユーザーは実測していないため必須にしない。未入力はnull保存原則どおり非表示）
- **設計思想**: みんカラのパーツレビュー文化（「マフラー交換した」を皆が好んで表明する）を構造化してカードのメインコンテンツにする。パワー数値の正確性を競うのではなく、**「何をどう改造したか」の表明欲求**を受け止める器にする
- **クラス判定（近い仕様フィルタ）**: 厳密な公平クラス分けは主張しない。主軸=**タイヤ区分**（Sタイヤ/ハイグリップラジアル/一般ラジアル — タイム影響が最大で万人が納得しやすい唯一の軸）、副軸=**改造度バッジ**（ライト/ミドル/フルチューン — 申告パーツカテゴリ数から自動推定）＋駆動方式。比較UIでは「近い仕様の車と比べる」フィルタとして提示 → P2の「不公平な比較で削られたくない」への直接回答
- **カードの二面性**:
  - 比較UI: 比較画面・共有ブラウズ・マーケット商品ページの冒頭に必ず両車のカードを並置。「タイム差2秒。ただし相手はSタイヤ+80ps」が一目で分かる
  - コンテンツ: カード単体をSNS共有可能な画像として出力。ビルドの進化でカードが更新される＝**車を育てるゲームの図鑑体験**
- **データモデル拡張**: 現状のVehicleSetupConfig（調整可否定義）に加え、VehicleProfile（改造パーツリスト/タイヤ区分/駆動方式、任意でパワー・車重）を新設。改造履歴はタイムスタンプ付き配列とし、柱Cの Mod ROI に接続
- **マーケット接続**: 出品データにカードが付くことで「どんな車のデータか」の不安が消え、購買転換率に直結。ショップはブランドカラーのカードでポートフォリオ化（P4）

### 柱C: ペルソナ別ジャーニーパッケージ

単一UIに全部盛りせず、ペルソナごとの「勝ち体験」をモード/ビューとして提供する。

1. **レースウィークエンドモード（P1）**: 練習→予選→決勝をひとつのイベントとして束ね、セッション間のセット変更とタイム/テレメトリ変化を時系列デブリーフ。クラス内デルタ（匿名参加者との中間statブック）
2. **ビルドジャーナル（P2）**: 改造履歴×タイム推移のタイムライン。「車高調導入 → 筑波 -1.1秒」を自動注釈。本質は**みんカラのパーツレビュー文化×ロガー証憑付きタイムの接続**（みんカラは「換えた」は書けるが「何秒速くなったか」は示せない）。費用は任意入力で、入力時のみ **Mod ROI（円/秒）** を表示。タイム変化には腕・気温・タイヤ摩耗も混ざるため「因果の証明」ではなく「参考注釈」として控えめに表示する。ショップへの送客・アフィリエイトの将来収益源にもなる
3. **ファーストラップ・アルバム（P3）**: 走行会終了時に自動生成される「今日のハイライト」（ベストラップ3Dクリップ＋成長グラフ＋称号バッジ）。ゼロ操作で成果物が届く体験
4. **プロポートフォリオ（P4）**: 供給者の実績・出品・レビューの公開ページ。データSEOの受け皿を兼ねる
5. **イベントモード（P5）**: 走行会単位のコード発行→参加者の当日リーダーボード＋一括デブリーフ。主催者向け無償提供でGTMを加速

### 横串: コンテンツフライホイール（デファクト化の機構）

```
走る → 記録する（NSM） → 自動でコンテンツ生成（リプレイ/カード/ハイライト）
  → SNS共有（透かし入り） → 新規ユーザー流入 → データが増える
  → ベンチマーク/ゴーストの価値が上がる → 買う理由・共有する理由が増える → 走る
```

- **すべての成果物に透かしとディープリンクを埋め込む**。ユーザーの承認欲求充足がそのまま獲得チャネルになる設計（Strava/Nike Runのモデルのモータースポーツ版）
- **データSEO**（既存GTM計画）との合流: 「筑波 ロードスター セットアップ」検索 → 匿名ベンチマークページ → クラス別カード一覧 → 登録
- **標準化戦略**: AIM/デジスパイス/NMEA対応済みの中立ハブとして「どのロガーでも、ここに入れれば比較できる」ポジションを固める。将来的に共有リプレイの埋め込みウィジェット（YouTube概要欄・ブログ貼付）を提供し、外部にコンテンツの根を張る

## 3. 品質分類（Kano推定）と優先順位

| 機能 | 分類 | 根拠 |
|---|---|---|
| 3Dリプレイ A-1（単独再生） | 魅力品質 | 誰も要求していないが見た瞬間にWow。ただし単独では飽きる |
| ゴーストバトル A-2 | **魅力品質→事業の核** | H3（支払い意思）を直接押し上げる唯一の仕掛け |
| 共有クリップ A-3 | 魅力品質＋成長エンジン | P3獲得・フライホイールの起点 |
| スペックカード（比較文脈） | **当たり前品質（比較機能にとって）** | これが無い異種混合比較は「悔しいだけ」で体験が破綻する |
| 自動クラス判定 | 性能品質 | 精度が高いほど満足が伸びる |
| Mod ROI / ビルドジャーナル | 魅力品質 | 世界初指標。P2のリテンション核 |
| レースウィークエンドモード | 性能品質（P1限定） | P1には必須級、他ペルソナには不要 |
| イベントモード | 性能品質（GTM） | 獲得効率に線形に効く |

**優先順位の論理**:
1. **スペックカード＋クラス判定を最優先**（当たり前品質の欠落は共有・マーケット体験全体を毀損する。データモデル拡張は後から遡及できないため早いほど良い）
2. **3Dリプレイ A-1→A-3**（獲得エンジン。段階B「間引きトレース永続化」と同一のデータ基盤に乗るため、既存ロードマップと自然に合流）
3. **ゴーストバトル A-2** はH3検証（アンカーデータ先行販売）と同時期に投入し、「ゴースト付きデータ商品」の購入率を素のデータと比較計測する
4. ジャーニーパッケージはP3（ファーストラップ・アルバム）→P1→P2の順（人口×実装コスト比）

## 5. リスクと対抗策

1. **バンドルサイズ**（3D合流時に適用）: three.js追加は既存2.8MB問題を悪化させる → リプレイはルート単位のdynamic import必須。導入時にコード分割（バックログ#1）を同時実施
2. **3Dが「見た目だけ」で終わるリスク**（3D合流時に適用）: 単独再生（A-1）で止めない。必ずA-2（ゴースト）まで到達させ、比較・購買文脈に接続することを完了条件にする
3. **改造申告の信頼性**: マーケットの信頼毀損に直結 → 「証憑ありタイム（ロガー由来）」と「申告スペック（自己申告）」のラベル分離で対処。パワー等の数値検証はしない（そもそも任意項目）
4. **Track Titanの日本進出**: 本プランのカード文化・クラス文化・走行会チャネルは日本のモータースポーツ文化への深い適合であり、翻訳では複製できない。先行してコミュニティの言語（クラス名・イベント名）をプロダクト内に取り込むこと自体が防壁
5. **開発リソース分散**: H3検証（最優先）を遅らせないこと。柱A/Bはいずれも「H3の購入率を上げる装置」として位置づけ、独立プロジェクト化しない

## 6. サブエージェント委任のためのワークパッケージ分割

実装委任時のモデル振り分け方針: 要件定義・デザイン・難問=Fable系 / 実装=Sonnet・Opus。

| WP | 内容 | 種別 | 依存 |
|---|---|---|---|
| WP-1 | VehicleProfile データモデル設計（改造パーツ申告リスト・タイヤ区分・改造度バッジ推定・zodスキーマ・匿名化仕様） | 要件定義→実装 | なし（最優先） |

## 決定事項（2026-07-02 ユーザー判断）

1. **3Dリプレイは保留**: 別プロジェクトで実施中。本プランは静止画ベースの共有コンテンツで先行し、3D成果は後日合流
2. **パワー実測の区別表示は不要**: パワー・車重は任意の自己申告。カードの主役は**改造パーツ申告リスト**（みんカラ的な「何をどう改造したか」の表明文化を構造化）
3. **クラス判定はAttack準拠にしない**: タイヤ区分を主軸、改造度バッジを副軸とし、「近い仕様フィルタ」として提示（厳密な公平性は主張しない）
