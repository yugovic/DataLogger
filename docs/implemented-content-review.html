実装済みコンテンツ レビュー

この資料は、モックではなく現在実装されているセットアップ記録画面を対象に、 UI/UX とモータースポーツ利用時のビジネス観点で指摘事項を整理したものです。 主対象はタイヤ空気圧、ダンパー設定、セッション情報、走行履歴カードです。

対象: 実装済みコンテンツ 主対象: CarSetup / BasicInfoTab / SetupCard 観点: UI/UX / 現場運用 / 継続利用 / 収益接続
2件

データ信頼性と誤操作に関わる項目です。現場での記録ツールとして信用を落とす可能性があります。

3件

ピット運用や比較導線に直結する項目です。継続利用率と有料価値の体感に影響します。

1件

一覧の再利用性とデータ資産化の弱さです。プロダクトの差別化を弱めています。

1. 日時文脈が保持されず信頼性が落ちる 2. コピー操作が破壊的で戻せない 3. ドライバー情報が実データ化されていない 4. モバイル現場利用に対してレイアウトが重い 5. 空気圧UIが判断支援まで届いていない 6. 履歴カードでセット比較できない

総評

現状の実装は「入力できる」「保存できる」までは到達していますが、 モータースポーツ用途で重要な「あとで再現できる」「次の調整判断に使える」「チームやショップで共有しやすい」 という価値までは十分に設計されていません。 特に、セッション文脈の保持、誤操作耐性、履歴比較の見せ方が弱く、プロダクトの継続利用動機と課金価値を削っています。

Critical

過去データを開いても、画面上は“今のセッション”に見える

何が起きているか

  • 画面上部の日時表示が常に現在時刻ベースです。
  • 保存時も `new Date()` をそのまま入れており、ユーザーがセッション日時を明示的に扱えません。
  • 過去データの読み込み処理はあるのに、編集画面上の日時文脈が復元されません。
CarSetup.tsx:663 CarSetup.tsx:176-181 CarSetup.tsx:302-377

なぜ問題か

サーキット走行データは日時とコンディションの紐づきが重要です。 編集画面で過去データを開いたときに“今日の走行”に見えると、後から見返した際の信頼性が落ちます。 これは単なる表示問題ではなく、履歴価値とデータ再現性を下げる問題です。

改善方向

  • セッション日時を state と保存モデルで明示管理する。
  • 閲覧時は保存済み日時を表示し、新規作成時のみ現在時刻を初期値にする。
  • 日時が固定なのか編集可能なのかをUIで明確にする。
Critical

「走行前→走行後にコピー」が破壊的操作なのに、確認もUndoもない

何が起きているか

  • 1クリックで4輪すべての `after` を `before` で上書きします。
  • 同時に差分も再計算され、実測後圧の情報が消えます。
  • 確認ダイアログ、トーストUndo、操作履歴のいずれもありません。
BasicInfoTab.tsx:90-105 BasicInfoTab.tsx:93-100

なぜ問題か

ピットやパドックでは短時間で入力するため、誤タップは現実的に起こります。 しかもこの操作は一部の値ではなく4輪を一括で壊すため、事故時の影響が大きいです。 記録ツールで最も避けるべきのは、ユーザーが気づかないまま実測値を失うことです。

改善方向

  • 初回は確認ダイアログを出す。
  • 直後に「元に戻す」を付けたトーストを出す。
  • 一括コピーではなく「空欄のときだけコピー」も選べるようにする。
High

ドライバー情報がUI上は存在するが、実データとして扱えていない

何が起きているか

  • セッションバーにはドライバー名らしき入力がありますが、固定初期値に近い扱いです。
  • 保存モデルに `driver` フィールドがありません。
  • 履歴や比較で「誰のセットか」を辿れません。
CarSetup.tsx:703-719 src/types/setup.ts:94-113 src/hooks/useSetupState.ts:73-85

なぜ問題か

個人利用なら見逃されることがありますが、ショップ運用やチーム共有では致命的です。 同じ車両でもドライバーが違えばフィードバックもセットの意味も変わります。 将来の共有機能や知見DB、データ販売にも接続しづらくなります。

改善方向

  • 保存モデルに `driver` を追加する。
  • 編集画面、履歴カード、フィルターで一貫して扱う。
  • チーム利用を想定するなら `driverId` と表示名を分ける。
High

モバイル現場利用に対して、レイアウトが横並び前提で重い

何が起きているか

  • タイヤ空気圧とダンパー設定が固定2カラムです。
  • ダンパー設定カード内もさらに2カラム構成です。
  • 環境データとタイヤ情報も狭い画面で2カラム前提です。
  • 数値入力コンポーネントも横幅を取りやすく、縦の視線移動より横圧迫が強いです。

なぜ問題か

実際の利用現場はデスクトップよりスマホが主になりやすく、片手操作や短時間確認が前提です。 横並びの情報密度が高いと、入力しづらいだけでなく、4輪の状態を一目で掴みづらくなります。 使いづらさは記録漏れと離脱に直結します。

改善方向

  • モバイルでは1カラム優先に切り替える。
  • タイヤ空気圧は4輪を縦に並べ、差分を右端または下段に整理する。
  • ダンパーは一覧入力より、前後単位またはFL/FR/RL/RRを切り替えて入力する設計も検討する。
High

空気圧UIが「記録」止まりで、「次の判断」に繋がっていない

何が起きているか

  • 見せ方の中心が `走行前 / 走行後 / 比較 / 差分` に留まっています。
  • 差分の色分けは符号ベースで、目標圧との関係を示していません。
  • 冷間目標、温間目標、次走行の調整案がUI上に存在しません。
BasicInfoTab.tsx:81-88 BasicInfoTab.tsx:155-164 SPECIFICATION.md:91-102

なぜ問題か

モータースポーツユーザーは、数値の保存そのものではなく、 その数値から「次に何kPa動かすか」を知りたいはずです。 現状のままでは記録帳としては使えても、判断支援ツールとしての価値が薄く、 AI提案や有料化への接続も弱いままです。

改善方向

  • 各輪に目標温間圧と目標との差を表示する。
  • 次走行に向けた推奨調整量を自動表示する。
  • 符号の色ではなく、目標レンジ内外で状態色を分ける。
Medium

履歴カードで“どのセットが速かったか”を比較しにくい

何が起きているか

  • 履歴カードには日付、サーキット、ラップ、天候、タイヤ銘柄は出ています。
  • 一方で、タイヤ圧やダンパー値などの中身は一覧上では見えません。
  • 速いセットを一覧比較するには、毎回詳細画面に入る必要があります。
SetupCard.tsx:64-154 SetupHistory.tsx:297-300

なぜ問題か

このプロダクトの価値は、蓄積した記録を比較して再利用できることにあります。 一覧で差が見えないと、ユーザーは「残してもあとで使えない」と感じやすくなります。 これは継続率だけでなく、将来のデータ資産価値にも悪影響です。

改善方向

  • 履歴カードに主要セット値のサマリーを出す。
  • 少なくともタイヤ圧の代表値とセッション条件は一覧で比較できるようにする。
  • ベストラップとの差分や前回比も表示対象にすると価値が上がる。

優先順位の提案

最初に直すべきなのは、1. 日時文脈の保持、2. コピー操作の安全化、3. ドライバー情報のデータ化です。 ここは記録ツールとしての信用に関わります。 その次に、4. モバイル前提の再設計と、5. 空気圧UIの判断支援化を進めると、 「ただ入力するだけのアプリ」から「次の走行に効くツール」へ一段上げられます。 最後に、6. 履歴カードの比較性を強化すると、蓄積データの価値がようやくユーザーに見える形になります。
