# テレメトリ解析モジュール（WP4: ロガー連携コア）

データロガーのファイルを取り込み、GPS 軌跡からラップタイムを自動抽出するための
純粋ロジック層。UI に依存しない（取込 UI は WP5）。

このモジュールが算出するラップタイムは、セットアップ記録の
**「ラップタイム証憑」（`lapTimeData.source = 'logger'`）の根拠**となる。
手入力タイムと異なり、実走 GPS ログから機械的に導出されたことを保証するのが
事業上の役割であり、そのため本モジュールは以下を厳守する:

- 欠損値の 0 変換・既定値充填の禁止（偽データ混入は事業上の欠陥）
- 不正フォーマットは「何として・なぜ」失敗したかを `TelemetryParseError` で明示
- 検出アルゴリズムの限界を本書に明文化（下記「精度と限界」）

## 公開 API（`src/lib/telemetry/index.ts`）

| 関数 | 役割 |
|---|---|
| `parseTelemetryFile(name, buffer)` | フォーマット自動判別 → パース（統一入口） |
| `detectFormat(name, buffer)` | 判別のみ（`'aim-csv' \| 'digispice-dtb' \| 'nmea'`） |
| `parseAimCsv / parseDigiSpiceDtb / parseNmeaRmc` | 各フォーマットのパーサー |
| `detectLaps(points, line, opts)` | ライン交差によるラップ検出 |
| `estimateStartFinishLine(points, opts)` | ライン未知時の自動推定 |
| `calcCumulativeDistance / calcLongG` | 累積距離（Haversine）・前後G |
| `guessTrack(points)`（`src/lib/tracks.ts`） | 軌跡からサーキット推定 |

典型的な取込フロー:

```ts
const session = parseTelemetryFile(file.name, await file.arrayBuffer());
const track = guessTrack(session.points);
const line = track?.startFinishLine ?? estimateStartFinishLine(session.points);
const result = line
  ? detectLaps(session.points, line, { minLapSeconds: track?.minLapSeconds })
  : { laps: [], bestLapIndex: null, crossingTimes: [] };
```

## 対応フォーマット

### 1. DigiSpice .dtb（バイナリ）

公式仕様非公開のため、実走データ（`src/components/demo/SampleData/`
`amuse_Z34_Ooi_0013_2_21_711.dtb`、鈴鹿フルコース 1 ラップ・5Hz・710 レコード）
のリバースエンジニアリングによる。

**レコード構造（128 バイト固定、リトルエンディアン）:**

| offset | 型 | 内容 |
|---|---|---|
| 0–7 | float64 | タイムスタンプ（Excel 序数日付 = 1899-12-30 起点の経過日数。JST 壁時計） |
| 8–15 | float64 | セッション先頭からの経過日数 |
| 16–31 | extended80 | 経過秒 |
| 32–47 | extended80 | 緯度（度） |
| 48–63 | extended80 | 経度（度） |
| 64–79 | extended80 | **高度（m）** |
| 80–95 | extended80 | **速度（km/h）** |
| 96–111 | extended80 | **逆方位（進行方位 + 180°）** |
| 112–127 | extended80 | 累積走行距離（km） |

`extended80` は x86 80bit 拡張倍精度を 16 バイト境界に配置したもの:
`bytes[0:8]` = 64bit 仮数（bit63 が明示的整数ビット）/ `bytes[8:10]` =
符号 1bit + 指数 15bit（バイアス 16383）/ `bytes[10:16]` = ゼロパディング。
JS の float64 へは仮数上位 53bit のみ反映される（相対誤差 ~1e-16、無視できる）。

**プロトタイプからの重要な修正:** TelemetryParserTest.html は
offset 64 を速度、offset 80 を方位と誤割当していた。実データ検証で確定:

- offset 80 の時間積分 = GPS 座標の Haversine 累積距離（5781m で一致）→ **速度**
- offset 64 は積分 3688m で距離と不一致、値域 67.8–107.7m が鈴鹿の標高
  プロファイル（高低差約 40m）と一致 → **高度**
- (offset 96 + 180°) と GPS 軌跡から計算した進行方位の差は中央値 0.83° → **逆方位**

誤割当のままだと速度レンジが 67–108km/h となり（実際は 52–228km/h）、
ラップタイム証憑に紐づく速度データが全て偽値になるところだった。

**ファイル単位の性質:** DigiSpice 公式ソフトはラップごとに .dtb を切り出して
出力する（ファイル名例 `…_0013_2_21_711.dtb` = ラップ 13・2:21.711。
サンプルの解析結果 141.711 秒と一致）。1 ラップ切り出しファイルでは
ライン交差は復帰時の 1 回しか観測できないため、`detectLaps` は OUT + IN
（NORMAL なし）を返す。これは「完全な周回を 2 回のライン通過で証明できて
いない」という正直な解釈であり、仕様である。複数ラップを含むセッション
ファイルでは通常どおり NORMAL ラップが切れる。

### 2. NMEA 0183 RMC（テキスト）

`$GPRMC` / `$GNRMC` を受理。XOR チェックサム検証（不一致はスキップして件数記録）、
ステータス A のみ採用、座標 ddmm.mmmm→度変換、ノット→km/h（×1.852）、
時刻巻き戻り検出による日跨ぎ対応（+86400s）。針路フィールドが空（停止中）の
場合は heading=null（0 への変換禁止）。RMC に高度は無いため altitude=null。

### 3. AIM CSV（Race Studio エクスポート）

`"キー","値"` 形式のメタデータ行 → チャネル名行 → 単位行 → データ行。
列名ゆらぎはプロトタイプ準拠（`GPS_Speed`/`GPS Speed`/`Speed`+単位km/h、
`GPS_Latitude`/`Latitude` 等）。単位行が無いエクスポートにも対応。
速度列が無い場合は GPS 座標差分から速度を導出する
（`meta.extra.SpeedDerivedFromGps = 'true'` を付記）。

## タイムゾーンの扱い

| フォーマット | ファイル内の時刻 | `meta.startTimestamp` |
|---|---|---|
| .dtb | Excel 序数日付（JST 壁時計） | JST→UTC 変換済みの正確な時刻 |
| NMEA | 規格上は UTC だが DigiSpice は JST を書き込む | UTC として解釈（DigiSpice 出力では実時刻と 9 時間ずれる可能性。経過秒・ラップタイムには影響しない） |
| AIM CSV | ロガー設定のローカル時刻 | JST と仮定して変換（国内利用前提） |

## ラップ検出アルゴリズム（`detectLaps`）

1. **前処理**: GPS 欠損・座標レンジ外・時間非増加の点を除去。直前の採用点
   からの移動速度が 150 m/s（540 km/h）を超える点は GPS 飛びとして除去。
2. **線分交差判定**: コントロールライン（2 点の線分）と、連続 GPS 点を結ぶ
   軌跡線分の交差を、ライン中点まわりの局所平面投影上で外積判定。
   交点がライン線分のパラメータ範囲 [0,1] 内にある場合のみ採用
   （延長線上の通過は無視）。点が厳密にライン上に乗った場合は正側として
   扱い、隣接 2 線分での二重計上を防ぐ。
3. **交差時刻の内挿**: 交差位置の線分内比率 s から
   `t = t_i + s·(t_{i+1} − t_i)` を算出（サブサンプル精度）。
4. **交差方向チェック**: 全交差の方向で多数決を取り少数派（逆走・切り返し）
   を除外。
5. **最小ラップ時間ガード**: 直前採用交差から `minLapSeconds`（デフォルト
   20 秒、トラック DB はコース毎に設定）未満の交差を破棄。ライン付近の
   GPS ノイズによる多重交差を吸収する。
6. **ラップ組み立て**: 最初の交差前 = OUT / 交差間 = NORMAL / 最後の交差後
   = IN。ベストラップは NORMAL の最速。

**ピットレーン等での低速横断の扱い（設計判断）**: 速度や所要時間で
弾くのではなく「ライン線分の長さをコース幅程度に保つ」ことで物理的に
交差させない設計とした。速度ガードは赤旗・コースイン直後など正当な低速
通過を誤って落とすリスクがあるため不採用。ライン定義がピットレーンを
覆ってしまっている場合に誤検出しうるのは既知の限界（下記）。

### スタート/フィニッシュライン自動推定（`estimateStartFinishLine`）

トラック DB に該当が無い場合のフォールバック。**精度より頑健性を優先**:

1. GPS 由来の対地速度（速度チャネル非依存）を中央値フィルタで平滑化し、
   **最高速度の点をアンカー**とする。最高速点は最長ストレート上にあり、
   (a) 周回ごとのレーシングラインの横ずれが最小 (b) 内挿精度が最良
   (c) 停止・徐行が起きにくい (d) 8 の字コース（鈴鹿等）の立体交差は
   低中速区間にあることが多く重なりにくい、という理由による。
2. アンカー半径 35m 以内の通過を時間クラスタリングで「パス」として抽出
   （= 軌跡の自己回帰性の確認）。**3 パス未満なら null**（周回と断定しない）。
3. 各パスの進行方位の円環平均から ±45° を超えるパスを除外（立体交差・
   逆走の防御）。残り 3 パス未満なら null。
4. 最接近点の重心をライン中心、平均方位の垂直をライン方向とし、横ずれの
   標準偏差から半長を決定（15–40m にクランプ）。
5. 推定ラインで `detectLaps` を実行し NORMAL が 1 本以上切れた場合のみ採用
   （自己検証）。

推定ラインはコース公式のコントロールラインとは一致しない（最高速地点
基準）が、**周回周期 = ラップタイムは同一**になる。証憑として使う場合は
「基準線は自動推定」である旨を UI で明示すること。

## 精度と限界

- **時間分解能**: 交差時刻は線形内挿によりサンプル間隔より細かい値を返す。
  誤差源は (a) サンプル間の軌跡を直線近似すること (b) GPS 位置誤差。
  5Hz・180km/h ではサンプル間 10m、ストレート上の直線近似誤差は実質ゼロで、
  位置誤差 ±3m が支配的 → **時刻誤差 ±0.06 秒程度**。1Hz・100km/h では
  サンプル間 28m となり、ライン付近がコーナーだと直線近似誤差が増える
  （±0.1〜0.3 秒）。検証: 実ラップを既知周期で 3 周連結した合成セッションで
  検出周期の誤差 < 1ms（回帰テストでロック済み）。
- **ラインを 2 回通過しないと NORMAL ラップは出ない**: 1 ラップ切り出し
  ファイル（DigiSpice のラップ別 .dtb）では OUT + IN になる。
- **トラック DB の座標精度**: `suzuka-full` のみ実走データ校正済み。他コースは
  航空写真知識からの近似値（誤差数十〜数百 m）であり、DB ラインで切れない
  場合は自動推定へフォールバックする運用を前提とする（`src/lib/tracks.ts`
  冒頭コメント参照）。
- **ピットレーンがコントロールライン延長上に近接するコース**では、ライン
  半長の設定次第でピット通過を交差と誤検出しうる。
- **GPS 多周回ドリフト**: 長時間セッションで受信状況が変わると、同一
  レーシングラインでも軌跡が数 m 平行移動することがある。ライン半長
  （±15〜25m）はこのドリフトを吸収するためのマージンでもある。
